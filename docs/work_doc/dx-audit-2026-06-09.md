# DX 实测审计：「一次适配、零改动换引擎」到底成不成立（2026-06-09）

方法：以真实应用开发者视角写实测代码（不看文档先写「天真代码」），对两条路径做行为验证。
实验脚本为一次性 scratch（已删除），构造了 5 个引擎：array-only、bytes/file-only、
fetchable_url-only、带必填 config（`api_key: SecretStr` + `default_language`）的云引擎、
以及已安装的真实 `dummy/echo` 与 `faster-whisper/whisper` 插件。

## 结论先行

- **路径 a（自动协商）成立。** 同一行应用代码 `engine.transcribe("meeting.wav")` 在
  array-only 与 bytes/file-only 引擎上零改动可用；URL-only 引擎按规范 fail-explicit
  （标准不是 upload-broker，设计如此）。错误信息总体质量高（裸 ndarray 的报错直接给出
  正确写法 `Pass AudioArray(samples, sample_rate)`）。
- **路径 b（配置穿透）在审计时不成立，现已修复。** 标准承诺 config 可被 UI 发现
  （G.3.1 / IC.3 secret 标记 / server 发布 provider_params schema），但 **引擎的 config
  类本身无法在不实例化的情况下获得**——而实例化恰恰需要待收集的凭证（循环依赖）。
  本次新增 `config_type` ClassVar + `registry.config_schema()` + `GET /v1/config-schema`
  关闭此缺口（详见「API 形状变更」）。

## 摩擦点（按严重度）

### F1 — 【严重，已修】config schema 在不实例化时不可发现（路径 b 的根阻塞）

mission 对 end user 的核心承诺（用户装引擎、在应用设置里配置）依赖应用能把
「用户自装引擎」的 init config 透出到自己的设置 UI。审计时唯一路径是「实例化后读
`type(engine.config)`」——对需要凭证的引擎是死循环。

**修复**（本次实现）：
- `EngineBase.config_type: ClassVar[type[BaseConfig[str]] | None] = None`（镜像
  `provider_params_type` 的先例）。
- `ModelRegistry.config_schema(name) -> dict | None`：经 `engine_class()` 读取，零实例化；
  `config_type` 非 BaseConfig 子类时 fail-loud（`FactoryLoadError`）。
- `GET /v1/config-schema/{model}`：wire 侧同构端点（镜像 params-schema；schema 只含字段
  形状不含值，无鉴权暴露与 capabilities 同理）。spec/server.md §3.6.1 已记。
- cookbook 两个插件已声明 `config_type`；合规套件：未声明 → warning（DX nudge）；
  声明了非 BaseConfig → error；实例 config 与声明类型不符 → error。

修复后的应用代码（10 行目标达成）：

```python
registry = discover_models()
schema = registry.config_schema("acme/large")      # None ⇒ 引擎未声明
values = my_app.render_settings_form(schema)        # secret 字段带 format=password/writeOnly
engine = registry.create("acme/large", **values)
```

### F2 — 【中，已修】`create()` 的异常契约文档与行为不符

docstring 声称 `FactoryLoadError: Factory failed to load or execute`，实际 factory 执行期
错误（pydantic `ValidationError` / `ConfigError`）**原样传播**。注意：**传播是对的**——
server 的错误映射依赖区分「unknown model → 404」与「config 错误 → 422」；包装反而是 bug。
已改 docstring 明确契约：factory 自身的错误不包装，并指向 `config_schema()`。

应用安全模式（对任何引擎都成立）：

```python
try:
    engine = registry.create(key, **user_config)
except (ValidationError, ConfigError):
    schema = registry.config_schema(key)   # 引导用户补配置
```

### F3 — 【中，已修】引擎作者的语言轴 bug 晚爆且爆在用户身上

引擎声明了 `selectable_languages` 但 config 缺 `default_language` 时：构造成功（IC.9 纯
`__init__`）、合规检查通过、**用户第一次 transcribe 才 ConfigError**。已在
`check_entrypoints`（instantiate 路径）加检查：EngineBase 引擎直接复用运行时
`_validate_language_config`（合规判定与运行时零漂移，含 membership/规范化）；structural
引擎做 IC.6 存在性检查。错误级 issue，作者在发布前即被拦截。

### F4 — 【低，Task 4 处理】url-only / storage-only 引擎的协商死角 hint 不可行动

与 round-3 R3-AUDIO-NEG-04 同根，修复合并到 Task 4。

### F5 — 【低，文档建议】secret 标记在字段重声明时静默丢失

引擎作者把 mixin 的可选凭证改必填时若写 `api_key: SecretStr`（裸注解、不用
`secret_field`），pydantic 的字段覆盖语义会**丢掉 `secret: true` 标记**与类定义期
SecretStr 强制。无明文泄漏（SecretStr 类型本身在 JSON dump 仍掩码、schema 仍带
`format=password`/`writeOnly`），但显式标记契约弱化。建议写入 adapter 指南：重声明凭证
字段必须用 `api_key: SecretStr = secret_field(default=...)`。未改代码（风险有界）。

### F6 — 【低，未修】无「模型卡」批量 helper

选择器 UI 需要逐 key 调 `engine_class()`（每个插件模块被 import）。3 行/模型可接受；
若未来插件目录（G.4.3）落地，可考虑 `registry.cards()`。**待决策。**

### F7 — 【接受】pydantic `ValidationError` 作为公共异常契约

config 契约本身就是 pydantic（G.3.1），核心依赖里就有 pydantic，字段级错误信息质量高。
不另设包装异常；已在 `create()` docstring 写明。

## API 形状变更与取舍（待你审查）

1. **`config_type` 放 `EngineBase` 而非 `StandardASR` Protocol。**
   取舍：放 Protocol 会让所有 structural 引擎立即不合 Protocol（破坏面大）；放 EngineBase
   + 默认 `None` + getattr 防御读取，与 `provider_params_type` 的既有先例完全一致。
   代价：structural 引擎可以不声明（仅得合规 warning）。**如果你想把它升为 Protocol
   必填成员（pre-release 可以激进），是一个可逆的后续决定。**
2. **合规级别：未声明 `config_type` = warning 而非 error。**
   理由：batteries 渐进采用；error 会把现存全部 structural 测试引擎判死。规范层面它
   目前是 SHOULD。**若要升 MUST（v1 前），改一行 level 即可。**
3. **`GET /v1/config-schema` 无鉴权暴露。**
   论证：schema 只描述字段形状（含 secret 标记），从不含配置值；与 capabilities/
   params-schema 同一可发现性原则（spec §3.1）。server 在 v1 本来就不接受 wire 构造
   引擎，收集到的值由操作者侧进程消费。
4. **`registry.config_schema()` 对垃圾 `config_type` 抛 `FactoryLoadError`**（而非返回
   None）：坏元数据 fail-loud 是本仓库一贯哲学；None 仅保留给「未声明」。

## 杂项观察

- `PreparedAudio` 是按 `kind` 判别的 union-ish dataclass：接受多形态的引擎必须按
  `prepared.kind` 分发——adapter 指南值得加一句（我的 scratch 引擎第一版就写错了）。
- 重复 `discover_models()` 对空 `model_name` 的 warning 每次发现都会重复打印（E6 里
  4 次）——极低优先级，未动。
