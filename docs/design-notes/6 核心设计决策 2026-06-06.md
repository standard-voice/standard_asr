## D1 — 音频输入抽象：判别联合 + 能力协商 + 透传

**我们选择了**：`transcribe`/`start_transcription` 接受一个**判别联合**音频输入 `path|bytes|(ndarray, sample_rate)|AudioUrl`；每个引擎用 `properties.accepted_input` 声明它接受哪些形态；标准层**仅在无交集时**做最小必要转换（happy path 直接透传，零转换），转换由 battery-included 的 `audio_loader`（可选 `[audio]` extra）承担。

**我们得到了**：
- 应用「手里是什么就传什么」（mp3 路径 / bytes / 数组 / URL），**云与本地不写两套代码**。
- 云文件引擎（OpenAI/EL）可**透传**原始文件/URL → 保真、不重编码、不撞 25MB/2GB 体积上限。
- 只吃数组的本地引擎仍可用（标准层把文件/bytes/URL 解码成数组）。
- 解码依赖留在可选 extra，核心不绑重依赖。

**我们牺牲了**：
- 引擎必须声明 `accepted_input`（轻量实现负担）。
- 标准层要维护「输入形态 × 引擎接受形态」的协商/转换逻辑。
- 存在**协商死角**：仅持本地数组的 app + 只接受服务端可拉取 URL 的引擎 → 无法合成 URL → 显式报错；必须可经 `accepted_input` 自动判定并体现在引擎 card / 文档。

**备选方案**：
- **仅数组**（现 prototype 的 `np.float32` 唯一契约）：否决——断裂于云 API（无法传 URL、需重编码、易超体积上限）。
- **仅编码字节为正规**：否决——把解码依赖塞进每个本地引擎，数组-native 引擎（Qwen/sherpa）平白吃 encode+decode 往返。

**受影响的利益相关者**：
- **应用开发者**：✓ 极简，传任意输入、零云/本地分支。
- **ASR 引擎开发者**：声明 `accepted_input`（小负担）；可只支持自己擅长的形态。
- **其他语言开发者**：REST/WS 层用文件/base64/URL，天然契合。
- **最终用户**：间接受益——更广的引擎兼容、更省的云传输。

> 实现备注：需补一个**数组→编码-bytes encoder**（扩展 `save_utils.nparray_to_audio_file` 到内存 `BytesIO`），供「app 给数组、引擎只吃文件」的情形。

> **✅ 审查后定稿（normative 见 `设计规范.md` §AI.1–AI.5、AI.10）**：判别靠**显式标签类型**（`AudioPath/AudioBytes/AudioArray/AudioUrl/AudioBase64`），裸类型仅作封闭便利 coercion，**裸 str 永不猜 URL/base64**；`accepted_input` 封闭枚举 + **穷举转换矩阵** + `IncompatibleAudioInputError` + `can_accept/negotiate` 调用前判定（死角进引擎 card）；encoder 内存 BytesIO/WAV-16/mono/**预检 max_file_size**；**`AudioUrl` v1 仅透传，标准不自取（SSRF）**；新增 properties `accepted_input/max_file_size/max_audio_duration/supported_input_formats`。**v1 砍**：标准自取 URL→本地、upload-broker、多声道/高位深 encode、base64-native（折成 coercion）。

## D2 — 采样率与 canonical 格式：16kHz mono 为默认，率随音频走

**我们选择了**：canonical 格式 = **16kHz mono**（in-Python 数组 `float32 [-1,1]`；编码/流式 wire = 16-bit signed LE PCM）。**采样率随音频一起传递**：`(array, sample_rate)`；文件靠头部自描述；**裸数组假定为 canonical 16k**。`sample_rate` **批次选填、流式必填**（裸流无头部）。重采样责任默认在**引擎侧/服务端**；标准层仅对「严格指定率且不自转」的引擎在输入不符时重采样。8kHz 电话与 24kHz realtime 作为**显式 capability**。

**我们得到了**：
- 一个安全默认（16k）——不懂的开发者完全不用关心采样率。
- 海量原生 16k 模型零摩擦（调查 `[[4 採樣率調查報告]]` 实证：16k 是压倒性事实标准）。
- **8kHz 电话不被盲目升采样**（阿里/Google 官方明确警告会掉精度）。
- **24kHz realtime**（OpenAI Realtime「Always 24000」）等例外可被 `required_input_sample_rate` 表达。
- 因多数引擎自重采样，**标准层重采样退化为边缘 fallback**（见 D3）。

**我们牺牲了**：
- 裸数组无法自测采样率 → 必须假定 16k 或要求 `(array, sr)`（对传裸数组者的约束）。
- capability 表面变大：`native_sample_rate / accepted_sample_rates / required_input_sample_rate / resampling(server|engine|standard|none) / channels / encoding`。

**备选方案**：
- **写死全局强制 16k**：否决——8k 电话升采样掉精度；realtime 24k 无法表达；裸数组无从可靠转换。
- **完全不标准化、全靠协商**：否决——失去「不懂就用 16k」这个最有价值的简单默认。

**受影响的利益相关者**：应用开发者（记一个数字 16k 即可）；ASR 引擎开发者（声明原生/接受率，多数复用已有自重采样）；其他语言开发者（wire format 明确）；最终用户（电话、分语种场景受益）。

> **✅ 审查后定稿（normative 见 `设计规范.md` §AI.6–AI.7）**：**裸数组无采样率 → strict 抛错 / best_effort 假定 16k 但必发 diagnostic，绝不静默**；采样率字段**归位 Properties**（复用既有 `supported_sample_rates/supported_channels/audio_dtype`，不另造平行词汇）= `native_sample_rate/accepted_sample_rates/required_input_sample_rate/wire_encodings`；**8k 电话 = entrypoint preset 选择**（接 GAP-20，禁升采样原生 8k）；**24k realtime = `required_input_sample_rate`**（典范「标准重采样」情形）。

## D3 — 重采样：核心 ad-hoc fallback + 警告，重活交给 `[audio]`

**我们选择了**：核心内置一个 **numpy-only 的 ad-hoc 重采样器**（从宽松许可证开源项目移植、非手写、非 GPL；或实现不可版权化的标准算法）；**未装 `[audio]` 而走到重采样时发 WARNING**（说明这是 ad-hoc 兜底，建议装 `[audio]`）；`[audio]`（scipy `resample_poly` 或 `soxr` 等）提供深度优化的高质量路径。

**我们得到了**：永不因缺依赖**硬失败**（battery-included DX）；核心不绑重依赖；质量/性能路径可选升级；多数引擎自重采样 → 实际很少触发兜底。

**我们牺牲了**：要维护一个 ad-hoc 重采样器；无 `[audio]` 时质量/性能次优（但有显式 warning + 它本就是边缘场景）。

**备选方案**：**严格走 `[audio]`、没装就硬报错**——否决（违背 battery-included DX 优先）；**核心绑 scipy**——否决（重依赖 + 扩大版本冲突面，见 D4）。

**受影响的利益相关者**：应用开发者（不会因采样率硬失败）；ASR 引擎开发者（不必自己处理）。

> **✅ 审查后定稿（normative 见 `设计规范.md` §AI.8）**：四值 `resampling` enum → **`self_resamples: bool`**（capability）+ 「false 且不符 ⇒ 标准必须重采样，否则定义错误」；**fallback MUST 抗混叠（FFT-based，禁裸线性/抽取）+ `resampled_with=fallback` diagnostic**；许可证 **clean-room FFT，禁 vendoring SoX/soxr(LGPL)**（soxr 仅作 `[audio]` 依赖）；流式热路径缺 `[audio]` 时会话建立即报错。

## D4 — numpy 依赖：核心宽兼容 1&2，绝不强制 2+，硬冲突靠进程隔离

**我们选择了**：核心声明 `numpy>=1.24`、**无硬上限**、只用长期稳定的 API 子集（`frombuffer/astype/clip/mean/fft/nan_to_num` 等，1.24→2.x 行为一致）→ 核心**同时兼容 numpy 1 与 2**。**不强制**引擎上 numpy 2+。真正冲突的插件（numpy1 vs numpy2）经 **out-of-process 隔离**（各自 venv，跑在 FastAPI server / 子进程后，应用经协议通信）。

**我们得到了**：
- 核心**永不成为冲突源**——FunASR(`numpy<2`) ∩ 核心(`>=1.24`) = `1.24–1.26`，可解。
- big tent：不排除仍绑 numpy 1 的重要引擎（如 FunASR）。
- future-proof：无硬上限 → numpy N 发布不会瞬间炸掉所有安装；纯 Python 用法**无 ABI 顾虑**（ABI 破坏只影响编译型 C 扩展，那是引擎自己 pin 的事）。
- 硬冲突有**诚实的出路**：进程隔离把「依赖地狱」变成部署选择，正是 server 工具链的价值之一。

**我们牺牲了**：核心代码受限于稳定 numpy API 子集（CI 需对 numpy 1.26 与最新 2.x 双测）；插件-插件硬冲突**无法在单进程内解决**（这是 Python 事实，非设计可消除）→ 需要进程隔离的部署成本。

**备选方案**：
- **强制所有 compliant 引擎 numpy 2+**：否决——排除 FunASR 等现存引擎；无法强制三方依赖；把版本钉进 10 年标准正是要避免的脆性。标准只固定 **numpy-float32-ndarray 这个类型**，不固定版本。
- **核心钉死窄范围（如 `>=2`）**：否决——主动制造冲突。

**受影响的利益相关者**：应用开发者（装插件少踩 numpy 冲突）；ASR 引擎开发者（不被强制 numpy 版本）；最终用户（更多引擎可用）。

> 事实核验：Python 3.13 移除 `audioop`（PEP 594），但**未移除 `wave`**；已在 Python 3.14.5 上实测 `wave` 正常往返且不依赖 audioop → 核心的 stdlib-WAV 路径在 3.13+/3.14 零依赖可用。
> 可选未来精化：将 numpy 设为核心**完全可选**（纯云 file/URL 流不产生数组 → numpy-free），numpy 由 `[audio]` 或数组引擎带入。

> **✅ 审查后定稿（normative 见 `设计规范.md` §DEP）**：**事实修正**——numpy 1.x 无 cp313 wheel，故 floor 改 **interpreter-conditional**（`numpy>=1.26; py<3.13` / `numpy>=2.1; py>=3.13`），原「1.24–1.26 可解」示例在 3.13+ 为空（正是隔离逃生舱的用武之地）；**无上限**；`clip`/`astype` 非"行为一致"→ 防御（先 clip 再 cast、禁 copy=False）+ **ruff NPY201 + 双版本 CI + nightly canary**；进程隔离首选 **subprocess+UDS+shared_memory**（非 FastAPI）+ `engine_id→endpoint` 寻址 + isolation shim(fast-follow)；新增 **`standard-asr doctor`** 冲突诊断。可选 numpy v1 不做。

## D5 — 能力声明系统统一（对应 §4.2 / GAP-16 / #2）

**我们选择了**：唯一的层级化能力系统 `capabilities.<域>.<特性>`（域含 `batch` / `streaming`），每个特性节点携带 `{supported, 约束/metadata}`（吸收原 feat_flag 的 metadata 思路）。**废弃** idea_docs 的平铺 `FeatureFlag` enum 与笔记 3 的独立 `feat_flag` dict；**去掉 per-feature `version`**。

**我们得到了**：单一真相源；strict/best_effort 机制能一致地挂在每个特性上；batch/streaming 分域天然；子对象（如 `capabilities.language.candidate_languages`）顺理成章；开放键 + 层级，比 enum 更能容纳未知新特性（10 年视角）。

**我们牺牲了**：比「单一 bool 集」更复杂的结构；需迁移 prototype 的 `FeatureFlag`。

**备选方案**：**平铺 enum**（idea_docs）——否决（无法表达分域 / 约束 / metadata）；**feat_flag dict 带 version**——否决（version 对兼容性无用，协议兼容是按大版本号全有或全无；笔记 3 已自我质疑）。

**受影响的利益相关者**：应用开发者（一处查所有能力）；ASR 引擎开发者（结构化声明）。

> **✅ 审查后定稿（normative 见 `设计规范.md` §C）**：**两层**——`DeclaredCapabilities`(ClassVar，免实例化可发现) + `effective_capabilities`(实例，只能收窄 `effective ⊆ declared`)；**封闭 mode 域** batch/streaming(保留 job)；节点三 archetype(flag/bounded/enum，每个可取 `.supported`)；**缺失=不支持(fail-closed) + 容忍解析未知键 + additive-within-major + `x_*` 实验空间**；`engine.supports("点.路径")` fail-closed 点查 + 统一 JSON；**blanket `metadata` 砍**(YAGNI)。**trade-off：`max_candidate_languages` 从 Properties 移入 `candidate_languages.constraints.max`**（见下方说明）。

## D6 — Runtime 参数：固定可移植标准集 + 受控逃生舱（§4.4 / GAP-12/13 / #4）

**我们选择了**：**固定的可移植标准集**（`language`/`candidate_languages`/`prompt`/`word_timestamps`… 由 capability 门控）+ **唯一受控逃生舱** `provider_params`（有类型、可发现、**明确标注非可移植**）。**ASR 作者不得新增顶层 runtime 字段**；引擎特有旋钮一律走 `provider_params` 的 schema。

**我们得到了**：可移植性（app 切引擎不改业务代码）+ 高级旋钮可暴露（engine dev）+ 诚实隔离（`provider_params` 标注「用了即锁某引擎」）+ 自动 UI 生成（`provider_params` 有 pydantic schema，满足 G.3.1）。一举化解规范原本「让作者定义更多参数」与「应避免自定义参数」的自相矛盾。

**我们牺牲了**：标准集的演进需要治理流程；`provider_params` 不可移植（但显式、可发现）。

**备选方案**：**引擎子类化 `BaseTranscribeOptions` 加顶层字段**（idea_docs）——否决（破坏可替换性）；**完全不给逃生舱**——否决（扼杀进阶功能、降低采用意愿）。

**受影响的利益相关者**：应用开发者（可移植）；ASR 引擎开发者（仍能暴露特性）；其他语言开发者（标准集映射干净）。

> **✅ 审查后定稿（normative 见 `设计规范.md` §R）**：`RuntimeParams` **封死**(extra=forbid，作者不得加顶层字段) + 合规测试；**v1 可移植集 = language / candidate_languages / word_timestamps(粒度枚举) / `guidance.prompt` / `guidance.phrase_hints`**；`provider_params` typed + **extra=forbid + always-raise(独立于 best_effort) + 类型绑引擎(swap 安全) + JSON-Schema-over-wire + auto-UI 标注「锁定 {engine}」**；**guidance 引导家族**(扁平字段+共享契约+capability 子树，命名 `guidance`，`prompt` 合并 OpenAI/Whisper/Qwen/火山 context，`phrase_hints` 无权重，**不对称未知处理 + opt-in 单向降级 + `x_*` 扩展**，见综合 §8)；`bias_resource`→Init Config；识别行为(verbatim/ITN/标点)reserve 为三态 flag。**trade-off：provider_params 错误永远抛、不走 best_effort**（见下方）。

## D7 — Init Config 基模型（§4.5 / GAP-03/08 / #5）

**我们选择了**：`BaseConfig` = 判别器 `engine` + **一组「相关才用」的可选标准字段**（`device`/`default_language`/`default_candidate_languages`…，**对所有引擎非强制**，云引擎不声明即可）+ **引擎声明字段**（含凭证 `api_key`/`base_url`/`region`）+ **凭证约定**（env 回退命名如 `STANDARD_ASR_<ENGINE>_API_KEY`，禁硬编码、禁入日志）。

**我们得到了**：云/本地统一——`use_device` 从「准强制」降为「可选标准字段」，云引擎无设备问题消解；多 artifact（sherpa 多 onnx + tokens）可容纳；凭证有处安放且安全。

**我们牺牲了**：「可选标准字段」的适用性取决于引擎，需明确文档说明。

**备选方案**：**`use_device` 准强制**（prototype 隐含）——否决（云 API 无设备）；**凭证进 `transcribe` 请求**——否决（调查收敛 #6：认证/region/resource id 不应进核心转写请求）。

**受影响的利益相关者**：应用开发者（配置界面可自动生成）；ASR 引擎开发者（可加自定义 init 字段）；其他语言开发者（config schema 可暴露）。

> **✅ 审查后定稿（normative 见 `设计规范.md` §IC）**：`engine` 判别器 = **entrypoint 派生 engine_id**（registry 查找路由、非宇宙 Union、检测重复名）；**凭证 `SecretStr` + 脱敏序列化 + REST write-only + secret 标记**，密 vs 端点(base_url/region)分两类；env 回退 normalization + 碰撞检测 + 优先级；**适用性 mixin**（字段出现⇒适用，统一 §C/§AI/IC）；**`default_language`：适用⇒必填或默认**（解与已定稿语言设计的矛盾，保 effective_language total）；init/runtime 边界 + tie-breaker（能按请求变⇒runtime）+ 模型选择=preset；多 artifact = nested submodel + 路径 helper；**lazy 纯度不变量**（__init__ 无 IO/网络）；`bias_resource` 住这里。

## D8 — 返回 shape 恒定（§4.6 / GAP-05 / #6）

**我们选择了**：`TranscriptionResult` schema **恒定**；capability/flag 只决定 optional 字段是否被**填充**，绝不改变返回的**类型/形状**。多通道用 optional `channels` 字段表达，`response_format` 不把返回变成字符串。

**我们得到了**：应用切引擎、改参数时返回类型稳定，可写稳定逻辑；显式 > 隐式。

**我们牺牲了**：放弃「按 `response_format` 直接返回纯文本/SRT/VTT」的便利（改为从标准结果再渲染）。

**备选方案**：**随 flag 变形**（OpenAI `response_format` / NeMo `return_hypotheses` / EL 多通道顶层 `transcripts[]`）——否决（调查反模式 #6，破坏可替换性）。

**受影响的利益相关者**：应用开发者（稳定 schema）；其他语言开发者（JSON schema 稳定）。

> **✅ 审查后定稿（normative 见 `设计规范.md` §TR）**：schema 恒定；**null 规则**(capability=支持；`None`=未请求；`[]`=请求但空)；`Segment`/`Word` **流批共享** + **时间=float 秒、原点=音频起点**；**多通道顶层 text=全通道时序合并**(非 channel-0)+`channels`+可选 `channel`；**speaker shape 现在就预留**(Segment 权威+Word，feature 延后)；核心 **`to_srt/to_vtt`** 取代 response_format 字符串(provider 格式仅 `extra` 透传)；**`session.result()`** 流式归约；演进=新字段 optional/None、major 内冻结、实体等 niche 走扩展命名空间不进顶层。

## D9 — Streaming 命名：Scheme A「会话」

**我们选择了**：`start_transcription(audio_format, params) -> TranscriptionSession`；`session.send_audio(chunk)` / `session.end_audio()`；迭代得 `TranscriptionEvent`（`.type ∈ partial | final | done | error`）。批量法 `transcribe` 与流式法 `start_transcription` 为**两个方法、两种返回类型**。

**我们得到了**：最低理解门槛（全是平实全词）；`start_transcription` ⟷ `transcribe` 对仗清晰；**「session」1:1 映射到要标准化的 REST/WS 资源**（`POST /v1/transcription-sessions`）；与批量 `TranscriptionResult` 同根，构成 `TranscriptionResult` + `TranscriptionSession` + `TranscriptionEvent` 协调三件套。

**我们牺牲了**：「session」略泛（可接受）。

**备选方案**：**B `Transcriber`**（名词最传神、`for event in transcriber` 最舒服，但起始动词偏弱）；**C `TranscriptionStream`**（保留业界最通用的 `stream`，换掉被否的 `open_` 动词）；**D `LiveTranscription`**（动词对仗最紧、最口语）。`send_audio`/`end_audio`/`partial`/`final` 已被业界（OpenAI Realtime SDK 等）验证为最佳，跨方案锁定。

**受影响的利益相关者**：应用开发者 / 其他语言开发者（REST/WS 资源命名自然）。

## D10 — Streaming 模型【DRAFT — PENDING INDEPENDENT REVIEW】

> 融合「架构审查」与「streaming 补充调查 `[[3 streaming 补充调查 2026-06-05]]`」的全部修正。**审查通过前不进 `设计规范.md`。** 待实测项见调查 §4。

### D10.1 接口与并发
- **两方法**：`transcribe(audio) -> TranscriptionResult`（批，永远此类型）；`start_transcription(...) -> TranscriptionSession`（任何「流式输出」场景；输入可增量、也可一次性灌完）。
- **全双工、async 优先**：send 与 receive 解耦并发。**session 拥有 pump**：`session.feed(audio_source)`（托管喂入 + 异常传播 + 生命周期），或手动 `send_audio()/end_audio()`。**sync 版由标准统一封装**（一个线程跑事件循环、session 拥有、`__exit__` 拆除）——ASR 作者只写 async。

```python
async with engine.start_transcription(audio_format=microphone_format) as session:
    session.feed(microphone())
    async for event in session:
        if   event.type == "partial": show_live(event.segment_id, event.text)
        elif event.type == "final":   segments[event.segment_id] = event.text
        elif event.type == "error" and not event.recoverable: raise event.as_exception()
```

### D10.2 事件模型：稳定前缀 frontier
- `TranscriptionEvent`：`type(partial|final|done|error)` + `segment_id`（**稳定 id**，非位置索引）+ `text`（该段当前完整文本）+ `stable_until`（已冻结字/词数，**左侧绝对不可变**）+ 可选 `words[]`/`start`/`end`/`stability`。
- `partial`：该段当前假设；`stable_until` 标已冻结前缀。
- `final`：该段到达端点（句界）。
- **归一为累积/replace**（每个 partial = 该段当前完整文本）；追加式引擎（OpenAI delta）由适配器内部累积。
- **简单 app 无视 `stable_until`**：`partial`→显示、`final`→按 `segment_id` 提交；**语音 agent** 才读 `text[:stable_until]` 作为可安全行动的前缀。
- **硬约束**：adapter 必须**保守**设 `stable_until`（宁可偏小）；冻结的前缀绝不回退（应对 Canary waitk/alignatt 的逐 chunk 重预测）。

### D10.3 两级终态
- `final`（不因**新音频**再变）vs `closed`（连**后处理重格式化/ITN/标点**都不再变）。引擎声明能保证到哪级（应对 AssemblyAI 重格式化、ElevenLabs committed 可改性未明）。

### D10.4 段生命周期（应对两遍重分段）
- 除 `revise(同 segment_id 再发)` 外，提供 `supersede(old_ids[] -> new_id)` / `merge` / `split`，让一次 final 重排能合并/拆分既有段且 id 可追溯（应对 WeNet U2++、Granite two-pass 改变句界）。**高级/可选**。

### D10.5 DSM 支持
- 事件携带独立于文本 `stable_until` 的**音频时间游标**（`audio_processed_until`）+ padding/心跳，表达「等证据 vs 静音 vs 时间推进」（应对 Kyutai/Voxtral 的 padding token）。

### D10.6 能力（正交 + 子结构）
- `supports_streaming_input` ⟂ `supports_streaming_output`。
- `StreamingCapabilities`：`emits_partials`、**`no_interim`档**（流式输入、仅段末出 final，无 partial——FireRedASR2S/SenseVoice+VAD）、`word_stability`、`timestamps`（native-frame-aligned / post-align / none——Qwen3 流式为 none）、`target_latency`（可配、**可中途调整**——Nemotron 4 档 / Voxtral 连续）、`right_context`、`reconnect`（seamless / lossy / unsupported——DSM 有状态模型偏 lossy）、`finality_level`（final / closed）。

### D10.7 生命周期 / 健壮性（标准定义，非交给适配器）
- **透明重连**（默认）：藏掉 provider 会话上限（Google ~5min、EL `session_time_limit_exceeded`）；`segment_id`/时间戳/语言跨内部重连**连续**；对 DSM 有状态模型可能 lossy（需声明）。
- `error` 事件 `(code, recoverable, retriable_after)`，**经事件流投递**；定义 post-error 会话状态。
- `send_audio` after `end_audio` → 报错；`end_audio` 幂等；`done` **必达带超时**；half-open 有界。
- **背压**：partial **按段合并**、final **永不丢**；发送侧有界缓冲、溢出显式报错。
- **时间戳**：原点 = 会话起点，秒，跨段与跨重连单调。

### D10.8 OpenAI Audio SSE 归位（诚实化）
- `streaming_output=true, streaming_input=false`：session 把 `send_audio` 缓冲到 `end_audio` 再发一次请求、流式收结果。**明确文档化「`end_audio` 前无 live partial；要实时 partial 须 `streaming_input=true`」**。
- 提供「灌完即收集」helper，让纯流式引擎（Qwen3 vLLM、sherpa online）也能满足批量 `transcribe()`。

### D10.9 核心 vs 高级切分
- **核心必备**（每个流式引擎）：`partial`/`final` + `text` + `segment_id`；`end_audio`；`error`。
- **高级/能力门控**：`stable_until` 前缀、`supersede/merge/split`、`closed` 级、edit-ops/diff（可选携带，应对 Granite NLE 编辑式重写）、音频时间游标、`target_latency` 配置、重连分级、`no_interim`。

### D10 trade-off
**我们选择了**：上述「全双工 session + 稳定前缀 frontier + 段生命周期 + 两级终态 + 正交能力」模型。
**我们得到了**：在所有已知引擎（DSM / 两遍重打分 / encoder-provisional / 云 WS / 增量库）上**无损表达**流式语义；语音 agent 拿到「可安全行动前缀」；简单 app 仍极简（渐进式复杂度）。
**我们牺牲了**：事件模型比「partial/final 二态」复杂（以「核心/高级切分」+「渐进式复杂度」缓解）；对 adapter 加「保守设 `stable_until`」「透明重连保持连续」等硬约束。
**备选方案**：**单一 partial/final 二态**（初版）——否决（AssemblyAI 重格式化、Soniox 逐 token、两遍重分段均无法表达，且累积/replace 丢失逐词冻结前沿）；**prototype `transcribe_stream(Iterable)->Iterator` + 薄 `StreamChunk`**——否决（半双工；无 segment id / stability / revision / error / done）。
**受影响的利益相关者**：应用开发者（简单场景简单、voice agent 有前缀）；ASR 引擎开发者（能如实映射各自语义、核心负担小）；其他语言开发者（事件 schema 跨语言）；最终用户（实时字幕 / 语音助手体验）。
**待审查 / 待实测**：ElevenLabs committed 段是否事后可改（决定 `closed` 语义）；GLM-ASR-2512 流式传输细节；Nemotron/Parakeet RNNT 是否严格 immutable。见 `[[3 streaming 补充调查 2026-06-05]]` §4。

> **✅ 审查后定稿（normative 见 `设计规范.md` §ST）**：**修 B1 事实错**——OpenAI Audio SSE 整段输入也在转写完成前发 `partial`；输入时序 ⊥ 输出时序（`streaming_input=false, streaming_output=true` 仍发 partial）。**解 M1**——`supersede(old_ids→new_ids)` **升为核心事件**（一个 op 覆盖 revise/merge/split；核心 reduce 含它，消除「忽略致陈旧段」矛盾）+ 状态机 + **冻结前缀跨 supersede 保留** + `closed` 不可被 supersede。`stable_until`=**codepoints**、绑 right_context、只增不减。新增 `progress` 事件 + `audio_processed_until` 游标（DSM 心跳）。**透明重连=「会话存活」非「app 无感」**（发 `reconnect(lossy,gap)` + session 拥有有界音频缓冲）。`feed` vs 手动 single-owner；sync 桥适配器契约 + 合规测试；`done` 必达带超时。**v1 砍**：运行时 latency 调整、edit-ops、无缝 DSM 重连、merge/split 三原语（→一个 supersede）、繁复 closed 机制。
