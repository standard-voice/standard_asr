# Standard ASR 差距分析 (Gap Analysis) 2026-06-05

> **Phase 1 交付物**。本文以当前已定稿的设计文档为基线，以三个验证基准（OpenAI Audio API / ElevenLabs STT / Qwen3-ASR）和大调查中全部引擎为参照，产出一份可操作、有优先级的差距清单与行动计划。
>
> **完成本 Phase 后暂停，等待确认再进入 Phase 2。**

## 0. 分析方法与基线说明

### 0.1 设计输入与验证基准

- **设计输入（求共性）**：`1 大規模 ASR 調查 2026-05-09.md` 中全部 17 个引擎 + `idea_docs/` 全部内容 + FireRedASR 笔记。
- **验证基准（精确驱动）**：
  | 基准 | 代表模式 | 本次暴露的关键面 |
  |---|---|---|
  | OpenAI Audio API | 商业云 REST + SSE | multipart 文件输入、鉴权、错误码、response_format、stream=true 的「批量输入 + 流式输出」 |
  | ElevenLabs STT | 商业云 REST + WebSocket realtime | 文件/URL/云存储输入、丰富进阶能力、realtime partial/committed 语义 |
  | Qwen3-ASR | 开源本地推理 | `from_pretrained` 生命周期、np/path/url/base64 输入、library incremental streaming |

> **本次基准与 2026-05-30 旧差距分析（Whisper 本地 + Qwen3）的关键差异**：旧分析的两个基准都是「本地 np.array 引擎」，未触及云 API。本次把 **两个云 API 设为基准**，直接暴露出「`np.float32` 唯一音频契约」「鉴权与 `use_device` 不适用于云」「错误码映射」等**旧分析系统性低估的张力**。本文据此重做。

### 0.2 基线定义（什么算"已定稿"）—— 这是本分析最关键的框架决策

按用户给定的文档层级：

- **Source of Truth（计入"已定稿"）**：`设计/设计规范.md`、`设计笔记和决策/1 language code 设计.md`、`设计笔记和决策/2 runtime_params 设计.md`、`设计笔记和决策/3 feature flag 设计.md`（草案）。
- **调查参考（事实依据，不计入"已定稿"）**：大调查、`idea_docs/`。

**`idea_docs/` 是 era-0 的先验实现草案**（git 历史 `add: idea docs from repo`，引用了 `cookbook/`、实际代码、`standard_asr.runtime.validate_audio_input` 等），用户在 `工作进度 2026-02-17` 中已"决定重新启动开发"并以更高严谨度重做。因此：

> **本分析把 `idea_docs/` 中的 `StandardASR` Protocol、`TranscriptionResult`、`Segment/Word`、`BaseProperties`、`BaseConfig`、`StreamingASR`、`FeatureFlag` 等视为「先验草案 (prior-art draft)」——它们降低对应缺口的设计复杂度（已有可借鉴的起点），但缺口本身仍属"未定稿"，且其中多处与新方向存在已知冲突（见 §4），不能直接采纳。**

这与旧差距分析的判断一致：旧分析同样把 `transcribe()`、`TranscriptionResult`、音频输入、引擎基类、streaming 列为"完全未设计"，尽管 `idea_docs` 有草案。

### 0.3 优先级与判断标准（采用用户定义）

| 等级 | 含义 | 判断标准 |
|---|---|---|
| **P0 阻塞** | 缺它则规范无法驱动任一基准的核心转写流程 | 基础请求/响应、音频输入、文本输出、流式协议 |
| **P1 重要** | 缺它则规范可用但明显残缺 | 语言检测、时间戳、词级信息、错误处理、鉴权 |
| **P2 可延后** | 增强功能，不影响核心可用性 | 翻译、说话人分离、自定义词汇表、高级格式选项 |

> 注意用户定义把 **鉴权 (auth) 与 错误处理 归在 P1**。因此本文把"引擎能被构造并发出一次请求/拿到文本"的**结构骨架**判为 P0，把"鉴权约定、错误体系"判为 P1——即便云 API 没有 key 实际无法调用，其结构容器（Init Config 允许引擎自定义字段如 `api_key`）已由 P0 的 BaseConfig 覆盖，缺的是约定与健壮性。

## 1. 验证基准能力矩阵（正向验证骨架）

对每个基准逐项检查"当前规范是否已覆盖"。状态图例：✅已定稿 / 🟡部分或仅概念 / ❌未设计 / 📄idea_docs有先验草案。

### 1.1 OpenAI Audio API（`/v1/audio/transcriptions`、`/v1/audio/translations`）

| 调用面 | 具体能力 | 规范覆盖 | 对应缺口 |
|---|---|---|---|
| 鉴权 | Bearer API key、base_url | ❌（Init Config 无凭证约定） | GAP-08 |
| 引擎构造 | SDK client（key/base_url/timeout） | ❌📄（无 BaseConfig 定稿） | GAP-03 |
| 模型选择 | `model`（whisper-1 / gpt-4o-transcribe / -mini / -diarize），请求级必填 | 🟡（与 entrypoint preset 映射未定） | GAP-20 |
| 音频输入 | multipart `file`（编码文件），25MB，mp3/mp4/m4a/wav/webm/flac/ogg/…，**不接受 URL/np** | ❌📄（现契约假设 np.float32） | GAP-04 |
| 语言 | `language` ISO-639-1，可省略自动检测 | 🟡（语言系统已定稿，但 ISO-639-1↔BCP-47 转换未设计） | GAP-17 |
| 核心响应 | `text` | ❌📄 | GAP-05 |
| 时间戳 | `verbose_json` → `segments[]`/`words[]`（秒） | ❌📄 | GAP-09 |
| 检测语言 | verbose 含 `language`、`duration` | ❌ | GAP-10 |
| 提示 | `prompt`（须匹配音频语言） | ❌ | GAP-12 |
| 采样温度 | `temperature` | ❌（应入 escape hatch 或标准集） | GAP-12/13 |
| 流式 | `stream=true` SSE：`transcript.text.delta` / `.done` / `.segment`；**整段音频一次上传、结果增量返回**（whisper-1 不支持，仅 gpt-4o-*） | ❌📄 | GAP-06/07 |
| 响应格式 | `response_format` = json/text/**srt/vtt**/verbose_json/diarized_json | ❌ | GAP-31 |
| 时间戳粒度 | `timestamp_granularities[]` = word/segment | ❌ | GAP-30 |
| logprobs | `include[]=logprobs` | ❌ | GAP-30 |
| 翻译 | `/audio/translations`（→英文） | ❌ | GAP-22 |
| 说话人 | `gpt-4o-transcribe-diarize` + `diarized_json`、`known_speaker_names/references[]` | ❌ | GAP-23 |
| 错误 | 标准 OpenAI 错误 + rate-limit 响应头 | ❌ | GAP-11 |
| 分块策略 | `chunking_strategy`（auto/server_vad） | ❌（可入 escape hatch） | GAP-13/26 |

> **OpenAI 暴露的"按 model 改变能力"问题**：同一 endpoint 下 whisper-1 与 gpt-4o-* 的可用 `response_format` / streaming / diarization / prompt 完全不同。这要求 **capability 必须是"模型级（即 entrypoint-preset 级）声明"**，而非"引擎级"——强化了"一个 model 一个 entrypoint"的设计，但也要求 capability 系统能表达这种差异（GAP-15/16）。

### 1.3 Qwen3-ASR（`from_pretrained` + `transcribe` + streaming state）

| 调用面 | 具体能力 | 规范覆盖 | 对应缺口 |
|---|---|---|---|
| 引擎构造 | `from_pretrained(model_path, forced_aligner, forced_aligner_kwargs, max_inference_batch_size=32, max_new_tokens=512, backend)` | ❌📄（无 BaseConfig；含 forced_aligner 这类子组件） | GAP-03 |
| 设备 | device/backend（transformers/vLLM） | ✅（`use_device` 已定稿） | — |
| 音频输入 | local path / URL / base64 / `(np.ndarray, sr)`；内部归一 mono 16k float32；ASR≤1200s，对齐≤180s | 🟡📄（np 部分有先验，URL/base64/长度上限未建模） | GAP-04/18 |
| `transcribe` | `transcribe(audio, context="", language=None, return_time_stamps=False)` | ❌📄 | GAP-02 |
| 语言 | `language`=语言名（如 "Chinese"），可 None→auto；30 语言+22 方言 | 🟡（语言名↔BCP-47 转换未设计） | GAP-17 |
| 上下文 | `context` 字符串 | ❌ | GAP-12 |
| 核心响应 | `list[ASRTranscription(language, text, time_stamps)]` | ❌📄 | GAP-05/10 |
| 词/段时间戳 | `return_time_stamps=True`（需 forced_aligner） | ❌（capability+param+result 三处联动未设计） | GAP-09/14 |
| 流式 | `init_streaming_state()`（仅 vLLM、单流、**不支持 timestamps**、默认 2s chunk）→ `streaming_transcribe(pcm16k, state)`（重喂全部历史、回滚近期 token 作 prefix）→ `finish_streaming_transcribe()` | ❌📄（library incremental，非网络协议） | GAP-06/07/15 |
| 错误 | Python `ValueError` | ❌ | GAP-11 |

### 1.4 三基准未覆盖、但大调查要求标准前瞻的重要模式

| 模式 | 来源引擎 | 标准启示 | 对应缺口 |
|---|---|---|---|
| submit/poll/webhook 异步 job | AWS/Azure/Google/腾讯/火山/阿里 | 必须建模"云存储 URI 必填 + 任务生命周期"，否则一大类云引擎无法适配 | GAP-32 |
| 强制云存储输入 | AWS batch(S3)、Google(gs://)、阿里(file_link) | 音频输入抽象须含"URI 引用"且引擎可声明"只接受 URI" | GAP-04 |
| provider-specific 模型码即语言 | 腾讯 `16k_zh`、`16k_multi_lang` | 语言映射层须容纳"语言绑定在 engine/model 名" | GAP-17/20 |
| interim + stability 数值 | Google `is_final+stability`、AWS `IsPartial+Stable` | 流式事件须能携带稳定度，而非只有 partial/final 二值 | GAP-21 |
| 多 artifact 本地初始化 | sherpa-onnx（多个 onnx + tokens） | Init Config 须支持多文件/多路径 | GAP-03 |
| 返回 shape 随 flag 变形 | NeMo `return_hypotheses`、OpenAI `response_format`、EL 多通道 | 标准须**禁止**返回 shape 随 flag 改变（恒定 schema + optional 字段） | GAP-05 |

## 3. 差距清单

> 字段：ID｜标题｜对应能力（引擎/参数）｜缺失原因（遗漏/推迟/冲突）｜优先级｜复杂度。
> 复杂度：L=Low / M=Medium / H=High。📄=idea_docs 有可借鉴草案（降低复杂度）。

### P0 — 阻塞核心转写流程（缺它则无法驱动任一基准）

| ID | 标题 | 对应能力 | 缺失原因 | 复杂度 |
|---|---|---|---|---|
| **GAP-01** | 引擎基类 / 生命周期接口 | 三基准的构造→暴露 properties/capabilities→transcribe→释放 | 遗漏（entrypoint 只解决"如何发现"，未定义 EngineClass 契约） | M 📄 |
| **GAP-02** | `transcribe()` 标准签名 | OpenAI `create()`；EL `convert()`；Qwen `transcribe(audio,context,language,return_time_stamps)` | 遗漏（规范聚焦校验与语言，未进入核心契约层） | M 📄 |
| **GAP-03** | Init Config 基模型（BaseConfig） | Qwen `from_pretrained(...)`；云 client(key/base_url)；sherpa 多 artifact；`use_device` 对云不适用 | 遗漏（仅设计了 3 个具体字段，无基模型/discriminator/"通用 vs 引擎声明字段"划分） | M 📄 |
| **GAP-04** | 音频输入抽象（**本次最关键张力**） | OpenAI multipart 文件(25MB)；EL 文件/`cloud_storage_url`/`source_url`；Qwen np/path/URL/base64；AWS/Google 强制云 URI | 设计冲突（现 `np.float32` 唯一契约**断裂**于云 API：无法传 URL、需重编码、可能超体积上限） | H 📄 |
| **GAP-05** | `TranscriptionResult` 核心容器 + 文本输出 | 三基准的 `text`；shape 须恒定（禁止随 flag 变形） | 遗漏（核心返回类型未定稿） | M 📄 |
| **GAP-06** | Streaming 接口（**两种流形态**） | OpenAI SSE=**整段输入+流式输出**；EL realtime=增量输入+输出；Qwen=library 增量 state | 遗漏 + 设计冲突（单一 `feed(chunk)` 模型无法覆盖 OpenAI"整段输入流式输出"） | H 📄 |
| **GAP-07** | Streaming 事件 / 结果模型 | partial/final/done/error；EL `partial`("may change")/`committed`；OpenAI delta/done/segment；Qwen 回滚 prefix | 遗漏（与 GAP-06 同源） | H 📄 |

### P1 — 重要（可用但明显残缺：语言检测/时间戳/词级/错误/鉴权）

|---|---|---|---|---|

| **GAP-10** | `detected_language` + `language_confidence` 结果字段 | OpenAI verbose `language`；EL `language_probability`；Qwen per-result `language` | 遗漏（TODO 已列但未设计） | L |
| **GAP-11** | 标准错误类型体系 + provider 错误映射 | OpenAI HTTP/rate-limit；EL `auth_error/quota_exceeded/rate_limited`；Qwen `ValueError` | 遗漏（仅 `UnsupportedFeatureError` 提名；云错误码是 OpenAI 验证重点） | M |

## 4. 关键设计张力（Phase 2 必须先决策的核心冲突）

这些不是单点缺口，而是**贯穿多个缺口的根本张力**，建议 Phase 2 开篇先做这 6 个顶层决策，再填具体缺口。

### 4.1 音频输入：`np.float32` 唯一契约 ⟂ 云 API 的文件/URL（GAP-04，最高优先）

- **现状**：idea_docs 契约"所有引擎必须接受 np.float32/16k/mono"。
- **断裂**：OpenAI 只收 multipart 编码文件（25MB）；EL 收文件 + `cloud_storage_url`/`source_url`（服务端拉取，≤2GB）；AWS/Google 强制云存储 URI。若强制 np.float32：① 丢失"传 URL 让服务端取大文件"的能力；② 适配器须把数组重编码为 WAV 再上传，长音频易超体积上限；③ 与"应用开发者友好/零配置"哲学冲突（应用本来手里就是个 mp3 文件）。
- **方向（待 Phase 2 定）**：音频输入应为**有判别的联合类型**（decoded array｜encoded bytes/文件｜URI 引用），引擎用 capability/properties 声明"接受哪些形态"，标准层只在双方都支持时做**最小必要转换**（如本地引擎需要 array 而应用给了文件 → 标准层解码；云引擎接受文件而应用给了 array → 标准层编码）。须遵循 trade-off 框架"泛化 > 单引擎"，且不能让应用为云/本地写两套代码。

### 4.2 能力声明三系统并存：Capability / FeatureFlag / feat_flag（GAP-16）

- 三套语义重叠的系统：`supports_*`（source-of-truth）、`FeatureFlag` enum（idea_docs）、`feat_flag` dict（笔记3）。必须收敛为**唯一**的能力声明系统，否则 strict/best_effort 机制、batch/streaming 分域、所有 P2 能力都无法一致地挂载。建议以 source-of-truth 的层级化 `capabilities.<domain>.<feature>` 为骨干，吸收 feat_flag 的 metadata 思路，废弃平铺 enum。

### 4.3 Streaming 两种形态 + partial 修订（GAP-06/07/21）

- **两种输入形态**：增量输入（EL realtime、Qwen）vs 整段输入+流式输出（OpenAI Audio SSE）。接口须同时容纳，可能是 `transcribe()` 返回值二态化（`TranscriptionResult` 或事件迭代器）+ 独立的增量输入会话接口。
- **streaming-input ⟂ streaming-output 正交**：idea_docs 已分 `STREAMING_INPUT`/`STREAMING_OUTPUT`，方向正确，须保留。
- **partial 语义**：须支持"interim 可被整体修订"（Google/AWS/EL）与"delta 追加"（OpenAI）两种，并能携带 stability（Google/AWS）。

### 4.4 Runtime 参数自相矛盾：标准集 vs 自定义（GAP-12/13）

- 规范同时写了"更多参数由 ASR 作者用 pydantic 定义"与"应避免 ASR 作者定义自己的 runtime 参数（否则 unswappable）"。必须裁决：**标准参数集（可移植）+ 受控 escape hatch（`extra`/`provider_params` 透传，且不进入可移植契约）**。否则要么扼杀进阶功能，要么破坏可替换性。

### 4.5 Init Config 的本地/云二元（GAP-03/08）

- `use_device`、模型权重路径假设本地推理；云 API 需要的是 key/base_url/region/timeout。BaseConfig 须区分"标准可选字段"与"引擎声明字段"，且**不得把 `use_device` 设为对所有引擎强制**。凭证作为"引擎声明字段 + 标准约定（env 回退命名）"。

### 4.6 返回 shape 恒定性（GAP-05）

- OpenAI `response_format`、NeMo `return_hypotheses`、EL 多通道都让返回 shape 随 flag 改变——这是反模式（调查#6）。标准须确立：**`TranscriptionResult` schema 恒定，差异通过 optional 字段表达**，绝不随参数改变返回类型。

## 6. 总结

| 维度 | 状态 |
|---|---|
| 已生产级 | 语言系统（含 candidate languages、effective 解析、strict/best_effort 框架） |
| 深但窄 | Capability/Properties 边界、runtime 策略——仅在语言切面展开 |
| 骨架缺失（P0） | 音频输入抽象、Init Config 基模型、TranscriptionResult、引擎基类/transcribe 签名、Streaming 接口与事件 |
| 基本空白 | 错误体系、进阶能力族、异步 job、tooling |
| 本次新发现（云 API 基准触发） | ① np.float32 唯一契约断裂于云；② 鉴权/错误码缺位；③ `use_device` 不适用于云；④ OpenAI"整段输入+流式输出"是第二种 streaming 形态；⑤ 返回 shape 随 flag 变形须禁止 |

**核心判断**：当前设计在**语言这一垂直切面**已达事实标准级深度，但**水平骨架（B/D/G/I）尚未展开**。与 2026-05-30 旧分析相比，本次因把两个云 API 设为基准，额外暴露了 6 个根本设计张力（§4），其中**音频输入抽象（GAP-04）是优先级最高、复杂度最高、最能体现"通用标准 vs 单引擎"的决策点**。

**建议路径**：阶段 0 先裁决 6 大张力 → 阶段 1 搭骨架（解锁全部 non-streaming）→ 阶段 2 补 streaming → 阶段 3 长肌肉。语言系统的严谨度是后续每个模块应对齐的范本。
