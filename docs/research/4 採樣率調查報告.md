# Standard ASR 採樣率調查報告：16kHz 是否為事實上的通用原生採樣率？

## TL;DR
- **是的，16kHz 是壓倒性的事實標準**：本研究逐一查證的「辨識用」ASR 模型中，其原生特徵前端（mel/fbank/wav2vec/直接吃波形）幾乎全部期望 **16kHz mono**。明確的非 16kHz 原生例外只有兩類：(1) **8kHz 電話模型**（騰訊 `8k_*`、阿里「8K ASR 電話通道」，以及 AWS/Azure/Google 對 telephony 8kHz 的原生路由）；(2) **audio-LLM 的生成型 codec tokenizer 用 24kHz**（Higgs-Audio v2、MiMo-Audio）——但即便如此，這些模型的「語意/理解（ASR）」分支仍回到 16kHz。
- **必須嚴格區分三個層面**：模型原生採樣率（多為 16kHz）≠ API 接受的採樣率（雲端 API 普遍接受 8k–48kHz）≠ 重採樣責任歸屬。本地庫（whisper、faster-whisper、whisper.cpp、sherpa-onnx、FunASR）一律在客戶端/引擎內用 ffmpeg/PyAV/librosa/內建 resampler 轉成 16kHz；雲端 API 多在服務端重採樣。
- **對 Standard ASR 協議的建議**：將 **16kHz / mono / 16-bit signed little-endian PCM** 定為 canonical 音訊格式；`sample_rate` 設計為「串流必填、批次選填」；把「原生 8kHz 支援」「最高接受採樣率」「標稱輸入採樣率（如 OpenAI Realtime 的 24kHz）」做成明確的 capability 聲明；預設由引擎側負責重採樣，並允許客戶端宣告「已是 canonical 格式」以跳過。

## Key Findings

### 1. 16kHz 原生佔比
在本報告以一手來源逐一查證的開源/自建 ASR 模型中，**僅有兩類非 16kHz 原生**（8kHz 電話模型、24kHz 生成型 codec tokenizer），其餘全部為 16kHz 原生。確認為 16kHz 原生的包括：OpenAI Whisper 全系列、faster-whisper、whisper.cpp、WhisperX、NVIDIA Parakeet（tdt/ctc/rnnt 全尺寸、v2/v3）、NVIDIA Canary 與 Canary-Qwen、IBM Granite Speech 3.3（2b/8b）、Cohere Transcribe、小紅書 FireRedASR（AED/LLM）與 FireRedASR2、阿里 SenseVoiceSmall、FunASR Paraformer 全系列、Qwen3-ASR、sherpa-onnx 所有官方預訓練模型、Moonshine（v1/v2）、Mistral Voxtral。這構成了「16kHz 是事實標準」的有力證據。

### 2. 非 16kHz 原生的例外（高價值發現）
- **8kHz 電話模型（真正原生 8kHz）**：騰訊雲 `8k_zh` 等 `8k_*` 引擎、阿里雲「8K ASR（電話通道）」模型。這些是用 8kHz 電話資料獨立訓練的模型。阿里官方明言：「8kHz 的電話音頻應直接使用 8kHz 模型，而不是升採樣到 16kHz 再識別」；Google STT 同樣警告「legacy telephony audio…use sample rates of 8000 Hz…provide the audio…at its native sample rate」（不要升採樣）。
- **24kHz audio codec tokenizer（屬 audio-LLM 的生成分支，非 ASR 辨識前端）**：
  - **BosonAI Higgs-Audio v2 tokenizer**：HuggingFace transformers `HiggsAudioV2TokenizerConfig` 明列 `sample_rate: int = 24000`、官方 tokenizer 卡「Unified 24 kHz Training」；但其語意分支 `semantic_sample_rate (defaults to 16000)`。官方部落格亦稱「we upgraded our audio pipeline from 16kHz to 24kHz」。
  - **小米 MiMo-Audio tokenizer**：一手來源確認為「1.2B-parameter Transformer operating at 25 Hz…eight-layer RVQ stack to generate 200 tokens per second」（HF 模型卡），patch encoder 將 25Hz 降採樣至 6.25Hz 供 LLM（arXiv:2512.23808）。其 codec 屬與 Higgs 同類的高採樣率生成型 tokenizer，但論文未逐字標明確切 Hz 數值（見 Caveats）。
  - **關鍵區分**：這類是「TTS/生成」用途的 codec，與「ASR 辨識前端」是不同的元件，不應混為一談。
- **接受高採樣率但前端仍是 16kHz 的特例**：Moonshine 直接吃 16kHz 原始波形（不用 mel）；Voxtral 前端為 16kHz、再以 12.5Hz 幀率下採樣。

### 3. 雲端 API 普遍接受多採樣率（服務端/客戶端重採樣）
- **OpenAI**：檔案 API（whisper-1、gpt-4o-transcribe、gpt-4o-mini-transcribe、gpt-4o-transcribe-diarize）接受常見壓縮格式，服務端處理。**Realtime/transcription**（`gpt-realtime-whisper`）標稱 **pcm16 @ 24kHz mono little-endian**——官方 API reference 逐字「The sample rate of the audio. Always 24000」、指南「Use 24 kHz mono PCM when sending audio/pcm」。這是「標稱輸入採樣率非 16kHz」的重要例外（部分範例可設 16000，但 24000 為預設/標稱值）。
- **ElevenLabs Scribe realtime**：音頻格式枚舉含 pcm_8000/16000/22050/24000/44100/48000 與 ulaw_8000；但 SDK 在 microphone 模式「always targets 16000 Hz PCM16 internally」，context 採樣率不同則送 worklet resample——即內部統一到 16kHz。
- **Amazon Transcribe**：batch 與 streaming 皆接受 `MediaSampleRateHertz`「Valid Range: Minimum value of 8000. Maximum value of 48000」；batch「In most cases, you can omit MediaSampleRateHertz and let Amazon Transcribe determine the sample rate」（選填、服務端偵測），streaming `StartStreamTranscription`「Required: Yes」（必填）。
- **Google Cloud STT v1/v2（Chirp/Chirp 2/3）**：`sample_rate_hertz` 接受範圍廣，WAV/FLAC 可省略；官方最佳實務「capture audio using a sample rate of 16000 Hz」，但對既有電話 8kHz 音頻明言不要升採樣、按原採樣率送。
- **Azure Speech**：SDK 預設 WAV 16k/8k 16-bit mono PCM；其他格式透過 GStreamer 解碼為 PCM；服務端內部一律轉 PCM；embedded 模式支援 8k/16k。
- **騰訊雲**：實時 WS 要求 16000 或 8000Hz；引擎碼 `16k_*` / `8k_*` 區分原生引擎；錄音文件識別支援 8K/16K。
- **阿里雲智能語音**：實時 WS 僅 8000/16000Hz、16-bit、mono PCM；錄音文件識別可加 `enable_sample_rate_adaptive=true` 做自適應。
- **火山引擎豆包**：文件識別宣稱「採樣率無要求」（少數派做法）；流式範例 `rate: 16000`。
- **Deepgram / AssemblyAI / Speechmatics**：皆提供可配置 `sample_rate`，並支援 8kHz mulaw 電話音頻（Deepgram 在設定 `encoding` 時 `sample_rate` 必填；Speechmatics `audio_format.sample_rate` 為必填整數；AssemblyAI 以連線 query 參數 `sample_rate`，預設 16000）。

### 一手來源關鍵證據摘錄
- **Whisper**：`whisper/audio.py` 中 `SAMPLE_RATE = 16000`；`load_audio` 用 ffmpeg `-ar {sr} -ac 1 -acodec pcm_s16le` 重採樣到 16kHz mono float32 [-1,1]。
- **faster-whisper**：`decode_audio(input_file, sampling_rate: int = 16000, split_stereo=False)`，內部 `av.audio.resampler.AudioResampler(format="s16", layout="mono", rate=sampling_rate)`，輸出 float32。
- **whisper.cpp**：`include/whisper.h` 內 `#define WHISPER_SAMPLE_RATE 16000`；README 要求 `ffmpeg -ar 16000 -ac 1 -c:a pcm_s16le`；runtime 錯誤「WAV file ... must be 16 kHz」。
- **sherpa-onnx**：`FeatureConfig.SampleRate`「It is 16000 for all pre-trained models provided by us」；`features.cc` 在輸入採樣率不同時 log「Creating a resampler: in_sample_rate: 8000 output_sample_rate: 16000」——確認引擎內做線性重採樣到模型期望採樣率。文件「sample rate 不必 16k」意指「可丟非 16k，引擎自動轉」，非「模型原生多採樣率」。
- **Granite Speech 3.3-8b**：模型卡程式碼 `assert wav.shape[0] == 1 and sr == 16000 # mono, 16khz`。
- **Canary**：`nvidia/canary-1b`「accepts single channel (mono) audio sampled at 16000 Hz」；`canary-qwen-2.5b`「16000 Hz Mono-channel Audio, Pre-Processing Not Needed」。
- **Cohere Transcribe**：`audio = load_audio("audio.wav", sampling_rate=16000)`；Conformer encoder 將波形轉 mel spectrogram。
- **Higgs-Audio v2 tokenizer**：`HiggsAudioV2TokenizerConfig` 之 `sample_rate: int = 24000` 與 `semantic_sample_rate (defaults to 16000)`；tokenizer 卡「Unified 24 kHz Training」。
- **MiMo-Audio tokenizer**：「1.2B-parameter Transformer operating at 25 Hz…200 tokens per second」（HF）；patch encoder 25Hz→6.25Hz（arXiv:2512.23808）。
- **OpenAI Realtime**：API reference「The sample rate of the audio. Always 24000」；指南「Use 24 kHz mono PCM when sending audio/pcm」，model `gpt-realtime-whisper`。
- **Amazon Transcribe**：`MediaSampleRateHertz ... Valid Range: Minimum value of 8000. Maximum value of 48000`；batch 可省略由服務端偵測，streaming Required: Yes。
- **Google STT**：「capture audio using a sample rate of 16000 Hz ... However, if your audio data has already been recorded at ... 8000 Hz ... do not resample your audio to 16000 Hz ... provide the audio ... at its native sample rate」。
- **阿里雲**：實時「支援的音頻採樣率：8000Hz/16000Hz」「16 bit、單聲道（mono）無壓縮的 PCM 或 WAV」；「8kHz 的電話音頻應直接使用 8kHz 模型」。
- **FireRedASR**：README「Convert audio to 16kHz 16-bit PCM format: ffmpeg -i input_audio -ar 16000 -ac 1 -acodec pcm_s16le -f wav」。
- **FunASR**：tutorial「specify the audio sampling rate fs (default is 16000)」；SenseVoice「16kHz sampling rate, single channel, 16 bit depth」。
- **Qwen3-ASR-Toolkit**：「Automatically converts audio from any sample rate and channel count to the 16kHz mono format required by the Qwen-ASR API」。

## Recommendations
1. **Canonical 格式定義（立即可定）**：採用 **16kHz, mono, 16-bit signed little-endian PCM** 作為 Standard ASR 的 wire-format 預設。理由：這是所有主流自建模型的原生格式，且為 ffmpeg/PyAV/librosa 重採樣的共同目標，跨 provider 對齊成本最低。
2. **`sample_rate` 參數設計**：批次（檔案）模式 **選填**（容器有 header 可自偵測，比照 AWS batch、Google WAV/FLAC）；串流模式 **必填**（比照 AWS streaming、阿里、騰訊、Speechmatics、Deepgram）。火山「無要求」是少數派，不宜作為標準。
3. **重採樣責任**：預設「引擎側負責重採樣」（對齊 sherpa-onnx/FunASR/Whisper 本地庫行為），並提供 capability flag 讓客戶端宣告「已是 canonical 16kHz」以跳過重採樣、降低延遲。
4. **Capability 聲明（關鍵設計）**：每個 provider/模型應宣告 `native_sample_rate`（如 16000 或 8000）、`accepted_sample_rates`（範圍或枚舉）、`required_input_sample_rate`（如 OpenAI Realtime 的 24000）、`resampling`（client/server/none）、`channels`、`encoding`。8kHz 電話模型應以獨立 `native_sample_rate=8000` 能力表達，而非 16kHz 模型的選項——因為它們是不同的訓練模型。
5. **8kHz 電話場景建模**：協議應允許 `sample_rate=8000` 並提供 `domain=telephony` 提示，路由到原生 8kHz 模型（若 provider 有），避免盲目升採樣造成精度損失（阿里、Google 官方均明確警告）。
6. **24kHz audio-LLM 例外**：capability 層必須能表達 `required_input_sample_rate=24000`（OpenAI Realtime），不可假設所有 realtime endpoint 都吃 16kHz。
7. **改變建議的門檻**：若未來主流 ASR 辨識前端普遍轉向 24kHz（目前僅生成型 codec 如此），或電話 8kHz 模型被 16kHz 通用模型全面取代（目前未發生），則應重新評估 canonical 格式。

## Caveats
- **Fish Audio ASR 採樣率未能由一手來源確認**：官方 docs（`POST /v1/asr`）僅說接受上傳音頻檔案，未明示採樣率要求；標記為「未找到/需進一步確認」。
- **Groq 採樣率為間接確認**：Groq 託管 whisper-large-v3 / -turbo，原生採樣率沿用 Whisper 的 16000，但 Groq 官方未獨立發佈特徵前端文件。
- **MiMo-Audio tokenizer 確切 Hz 數值未逐字證實**：一手來源確認 25Hz 幀率、8-layer RVQ、200 tokens/s，但論文未逐字標明 codec 的取樣 Hz 數值（推斷與 Higgs 同屬高採樣率生成型 codec，但標記為需進一步確認）。
- **Zoom Scribe**：未找到可核對其後端 ASR 權重或 config 的一手來源以確認採樣率；Zoom AI Companion 背後 ASR 未開源。標記為「未找到」。
- **NVIDIA Nemotron Speech Streaming**：已有公開 speech 模型卡（`nvidia/nemotron-speech-streaming-en-0.6b`，見 `[[3 streaming 补充调查 2026-06-05]]`）可確認產品存在；惟其採樣率規格未在模型卡／文檔明示，標記為「採樣率未明示」（而非「未見 speech release」）。
- **OpenAI Realtime 24kHz 的版本差異**：24kHz 來自最新 GA API reference（"Always 24000"），但部分舊 beta 範例使用 16kHz downsample；實作時應以最新 GA 文件為準。
- **二手來源限制**：部分「服務端內部重採樣」行為（如 Azure GStreamer→PCM）依官方文件描述推得，未逐一以原始碼確認，但與多方官方文件一致。
- **層面混淆風險提醒**：本報告反覆強調的核心——「API 接受多採樣率（服務端重採樣）」與「模型本身原生多採樣率」是兩件完全不同的事。雲端 API 接受 8k–48kHz 不代表其後端模型原生支援這些採樣率；絕大多數情況下後端仍是 16kHz（或 8kHz 電話）原生模型加上一層重採樣。設計標準時不可將兩者混為一談。
