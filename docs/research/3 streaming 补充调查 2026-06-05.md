# Standard ASR 流式行为补充调查 — 开源/开放权重引擎（第二批）

访问日期：2026-06-05

本笔记延续 `1 大規模 ASR 調查 2026-05-09.md`，**只聚焦"流式 / 实时"行为**，补查上轮未覆盖或只部分覆盖的引擎，开源 / 开放权重为主。结论只采纳已打开的官方文档 / 模型卡 / 仓库 README / 论文；凡未在这些来源确认的，标注"未确认 / 文档缺失"。本批最重要的发现是一类全新架构——**Delayed Streams Modeling（延迟流建模）**（Kyutai STT、Mistral Voxtral Realtime），它把"流式输出"建模为"文本流与音频流同帧率、逐帧发一 token、无证据时发 padding token"，输出严格 append-only，与上轮云厂商的"interim 可改写 + final"范式正交。详见末节"对标准的启示"。

## 2. 逐引擎详述

### NVIDIA Nemotron Speech Streaming 0.6B（新增，开放权重）

来源：模型卡 https://huggingface.co/nvidia/nemotron-speech-streaming-en-0.6b（2026-06-05）。

- **架构**：Cache-Aware FastConformer 编码器（24 层）+ RNNT 解码器。核心是"只处理新音频 chunk、复用缓存的 encoder context"，消除重叠重算。
- **延迟档位**：以 80ms 帧为单位，4 个可运行时切换的工作点 —— `0.08s [70,0]` / `0.16s [70,1]` / `0.56s [70,6]` / `1.12s [70,13]`，左上下文固定 70 帧，右上下文 0–13 帧；模型卡称这是"Dynamic Runtime Flexibility，在延迟–准确率 Pareto 曲线上选最优点"。**注意：右上下文 = 算法 lookahead，是延迟的来源。**
- **输出修订**：模型卡未明示 interim 是否被改写。RNNT + cache-aware 的常规行为是逐 token 追加、已发 token 不回退（≈ immutable 前沿），但**未逐字确认**。
- **时间戳**：支持 `enable_word_time_offsets`（word-level）。
- 与上轮 Parakeet 的关系：Parakeet 也有 `nvidia/parakeet-unified-en-0.6b`（offline+streaming 联合训练，宣称延迟低至 **160ms**）和 `nvidia/multitalker-parakeet-streaming-0.6b-v1`（多说话人流式），均印证 NVIDIA 把"同一 FastConformer 编码器在 offline 与 streaming 双模式联合训练"作为主线（HF 模型卡）。

### NVIDIA Canary — chunked & streaming decoding（新增细节）

来源：NeMo 文档 https://docs.nvidia.com/nemo-framework/user-guide/latest/nemotoolkit/asr/streaming_decoding/canary_chunked_and_streaming_decoding.html（2026-06-05）。

Canary 是 FastConformer 编码器 + Transformer 解码器的多语种 multitask（25 EU 语言 ASR + 翻译）。其流式有两条独立路径，且呈现一个**全新的"逐 chunk 重预测"模式**：

- **Chunked**：把长音频切成 `chunk_len_in_secs`（默认 40s）不重叠段，逐段独立识别后拼接 —— append-only、跨 chunk 不改写；`timestamps`（默认 False）**仅此模式支持**。
- **Streaming**：自适应逐 chunk 解码，**"每个 chunk 内可对同一位置 token 预测多次"**，由解码策略决定：
  - **Wait-K**：每个新音频 chunk 只发 1 个 token，要求 **无限左上下文**；准确率优先、延迟高（`decoding.waitk_lagging` 默认 2）。
  - **AlignAtt**：按 cross-attention 条件发 token，可一个 chunk 发多个 token，延迟更低、面向生产（`decoding.alignatt_thr` 默认 8）。
  - 参数：`chunk_secs`(2.0) / `left_context_secs`(10.0) / `right_context_secs`(2.0) / `decoding.hallucinations_detector`(True) + prompt 控制（`+prompt.task/source_lang/target_lang/pnc`）。
  - **streaming 模式文档未提 timestamps**。
- **对标准的关键点**：encoder-decoder + 流式策略意味着输出不是"单调增长的稳定前缀"——token 可能在 chunk 内被多次发出/替换，这比"前缀 + stable_until"更接近"整段 hypothesis 被反复重算"。详见末节。

### Kyutai STT（Moshi 家族，Delayed Streams Modeling）★ 新架构

来源：https://kyutai.org/stt ；仓库 https://github.com/kyutai-labs/delayed-streams-modeling ；论文 DSM 2509.08753（2026-06-05）。

- **机制（DSM）**：把"流式 X→Y"统一为多流序列建模。文本流相对音频流**延迟 N 帧**，给模型 lookahead；文本流做 padding 使时序与音频对齐。模型逐帧产出（要么一个文本 token，要么 padding），因此**输出是逐 token 单调前沿，已发即定（immutable）**。
- **模型 / 延迟**：`kyutai/stt-1b-en_fr`（0.5s 延迟，内置 **semantic VAD**，检测用户是否在说话——面向语音 agent）；`kyutai/stt-2.6b-en`（2.5s 延迟，英文，精度优先）。
- **时间戳**：**返回 word-level timestamps**，且因文本与音频帧对齐，时间戳是"免费"副产物（与传统"先转写再 forced-align"不同）。
- **服务化**：Rust WS server，L40S 上 64 路并发 @RTF 3x；H100 可 400 路实时流。
- **端点 / EOI**：semantic VAD 提供说话边界；WS 协议消息格式与显式 EOI 信号在 README 未逐字说明（**未确认**）。

### Mistral Voxtral Realtime（Voxtral-Mini-4B-Realtime-2602）★ 新架构，开放权重

来源：模型卡 https://huggingface.co/mistralai/Voxtral-Mini-4B-Realtime-2602 ；论文 https://arxiv.org/abs/2602.11298 ；纯 C 复刻 https://github.com/antirez/voxtral.c（2026-06-05）。

- **机制（DSM 系）**：causal audio encoder → MLP adapter 时间下采样到 **12.5Hz** → text decoder 自回归，audio 帧与 token 帧同为 80ms，**decoder 每帧预测 1 个 token**；**无足够声学证据时发 padding token `[P]`**（论文原文）。新增 causal encoder + Ada RMS-Norm 用于"delay conditioning"，端到端训练而非对离线模型加滑窗。
- **延迟**："1 token = 80ms"，可配 80ms–2.4s，**推荐 480ms**（此延迟下精度≈ Whisper）。13 语言。
- **输出修订**：**append-only、immutable** —— antirez C 复刻确认"token 一旦发出不修订"，`vox_stream_get()` 取"已生成 token"；`-I` 仅控制 encoder 批音频频率（影响 token 何时可取，不改 80ms 基础延迟）。
- **EOI**：必须显式 `vox_stream_finish(s)` 加 padding 并 flush 尾部 token。
- **部署**：vLLM Realtime API（WebSocket 音频会话）+ 开放权重 + ONNX 社区版。
- **与批模型区分**：`Voxtral Mini Transcribe V2` 是独立的离线模型（≤3h/请求），不要混淆。

### ElevenLabs Scribe v2 Realtime（深化上轮）

来源：https://elevenlabs.io/blog/how-scribe-v2-realtime-works ；https://elevenlabs.io/realtime-speech-to-text ；API 参考 https://elevenlabs.io/docs/api-reference/speech-to-text/v-1-speech-to-text-realtime（2026-06-05）。补充上轮已从 SDK 确认的事件集合。

- **两类转写**：`partial_transcript`（"live"，**明确 may change**，可被改写）→ `committed_transcript` / `committed_transcript_with_timestamps`（定稿；后者带 word-level ts + 检测语言）。
- **predictive transcription**：主动预测下一个词与标点以提升实时准确率——意味着 partial 里会出现"预测但随后可能改变"的内容（如 "I scream" → "Ice cream" 例子）。
- **commit 策略**：`manual`（默认，客户端控制何时 commit）或 `VAD`（静音阈值自动 commit）。官方表述："committing 会清空已处理的累积转写、起一个新段，但**不丢失上下文**。"
- **段的不可变性**：文档**未明确承诺** committed 段事后不再被后续输出修订（**未确认**——这是与我们标准模型直接相关的关键缺口）。
- **延迟**：宣称 ~150ms。session 最大时长 / chunk size 数值上限仍只在错误事件枚举里出现（上轮 SDK 已记录 `session_time_limit_exceeded` / `chunk_size_exceeded` 等）。

### IBM Granite Speech 3.3 / 4.1（批，非流式；但 NLE 编辑模式值得记）

来源：论文 https://arxiv.org/pdf/2505.08699 ；HF ibm-granite/granite-speech-3.3-8b ；IBM 公告；MarkTechPost（4.1）（2026-06-05）。

- **架构**：speech encoder（16 conformer blocks，CTC，**4s 音频块的 block-attention**，self-conditioned CTC）+ LoRA audio adapter + Granite LLM；**two-pass**（第一遍转写，第二遍可精修），整段处理。可接受任意长度输入（H100 上从容处理 20min），不受 Whisper 30s 窗口限制。
- **流式**：**不支持**。论文无流式声明；two-pass 是离线后处理精修，不是在线改写。
- **Granite Speech 4.1 NLE（Non-autoregressive LLM-based Editing）**：双向 LLM **一次前向**对 CTC 假设的所有位置同时预测编辑动作（copy / insert / delete / replace）。这是**批式纠错/重写**，非流式，但其"对已有假设做结构化编辑"的形态对我们"post-final reformatting revision"有直接启示（见末节）。

### Meta Omnilingual ASR（omniASR，批）

来源：论文 https://arxiv.org/html/2511.09690v1 ；HF facebook/omniASR-CTC-7B / omnilingual-asr 仓库（2026-06-05）。

- 共享 wav2vec2.0 编码器（300M/1B/3B/7B），三个 ASR 家族：CTC（线性层 + 字符级 CTC）、LLM ASR、zero-shot LLM ASR；**1600+ 语言**，Apache-2.0。
- **流式**：**无文档**。CTC 整段解码、LLM 自回归整段；CTC RTFx 极低（300M 在 A100 上 30s 音频 batch1 RTF≈0.001）但仍是离线。CTC 理论上可流式（逐帧 argmax），但**官方未提供流式推理路径**——标注未确认。

### Cohere Transcribe（cohere-transcribe-03-2026，批）★ 验证 Cohere 确有 ASR

来源：HF CohereLabs/cohere-transcribe-03-2026 ；Cohere 博客 https://cohere.com/blog/transcribe ；TechCrunch / VentureBeat（2026-06-05）。

- **存在性确认**：Cohere 2026-03 发布了**专用 ASR**（2B，audio-in/text-out，Apache-2.0，14 语言，WER 5.42%）。可经 API 或 Model Vault。
- **流式**：**不支持**。模型卡明确长音频靠 feature extractor 的 `max_audio_clip_s` 自动切块、逐块识别后**重组为最终输出**，不发 partial；所有示例都是整文件。55min 音频 RTFx≈525，是离线快速批处理。

### Xiaohongshu FireRedASR / FireRedASR2S（ASR 离线；VAD 流式）

来源：论文 https://arxiv.org/pdf/2501.14350 与 https://arxiv.org/pdf/2603.10420 ；GitHub FireRedTeam/FireRedASR(2S)（2026-06-05）。

- **FireRedASR-AED**（attention encoder-decoder）/ **-LLM**（Encoder-Adapter-LLM）：面向短视频/直播/语音输入/歌词，**整段离线解码**，支持 batch；中文 SOTA。**ASR 模块本身非流式**。
- **FireRedASR2S（"all-in-one"）**：把 ASR + VAD + LID + Punc 串成 pipeline。**FireRedVAD**（0.6M，DFSMN）**显式支持 streaming VAD / non-streaming VAD / mVAD**；但流水线里 ASR 仍是被 VAD 切段后**离线识别每段**。这是"流式 VAD + 离线 ASR"的典型工业拼装，**不是流式 ASR 解码**。

### Alibaba SenseVoiceSmall（明确非流式，纠正上轮模糊点）

来源：论文 https://arxiv.org/html/2407.04051v1 ；GitHub FunAudioLLM/SenseVoice ；Issue #143（2026-06-05）。

- 非自回归端到端，**整段一次推理**（10s 音频 70ms，比 Whisper-large 快 15x）。论文明言"**SenseVoice is not designed for streaming transcription**，未来工作才考虑可流式版本"。GitHub Issue #143 证实**无 cache-aware streaming 代码路径**。
- 上轮在 sherpa-onnx 章节里 SenseVoice 是作为 OfflineRecognizer 出现的，本批明确：**SenseVoice 只能"伪流式"（外接 VAD 切段后离线识别），不是真流式**。

### Xiaomi MiMo-V2.5-ASR / MiMo-Audio（开放权重为批；流式宣传存疑）

来源：HF XiaomiMiMo/MiMo-V2.5-ASR ；GitHub 同名仓库 ；Xiaomi 产品页 mimo.xiaomi.com ；gizmochina 报道 ；MiMo-Audio 论文 2512.23808（2026-06-05）。

- **MiMo-V2.5-ASR（已发布权重）**：仓库与模型卡**只提供离线接口** `model.asr_sft("audio.wav")`，无 streaming API、无 timestamps 文档。支持普通话/英文/吴语/粤语/闽南语/四川话等及中英混读，`audio_tag="<auto>"` 自动检测语言。
- **冲突记录**：部分二手报道与 Xiaomi"full voice pipeline"宣传称 ASR 支持"batch + real-time streaming"，但**开源版未见任何流式代码或文档**。结论：**开放权重 = 批**；流式可能仅存在于 Xiaomi 内部/云产品，未开源。标注存疑。
- **MiMo-Audio 7B**：audio-LLM，interleaved text/audio token + next-token 目标，偏 speech continuation / 多说话人生成，**非转写产品**；流式属性未确认。

### Moonshine v2（Useful Sensors）★ encoder 端 provisional 覆盖

来源：论文 https://arxiv.org/html/2602.12241v1（"Moonshine v2: Ergodic Streaming Encoder ASR"）；HF UsefulSensors/moonshine-streaming-tiny ；v1 论文 2410.15608（2026-06-05）。

- **机制**：**ergodic 滑窗 encoder**（bounded local attention，**无位置编码**）+ adapter 注入位置信息 + 标准自回归 decoder。外层窗 `(16 left, 4 right)`，中层 `(16 left, 0 right)`；`w_right=4 ⇒ 80ms 算法 lookahead`，注意力 O(T²)→O(Tw)。
- **输出修订（关键）**：论文给出明确的 **provisional → finalized** 模型 —— "可以从 *provisional*（尚未定稿）encoder 状态解码：最新后缀可能不准，但随更多音频到来，**provisional 状态被 finalized 状态覆盖，显示的转写自然变好**。"finalized 需 ~**320ms（16 帧）未来上下文**。
- decoder 仍逐 token 串行；论文未承诺 word-level timestamps。
- **意义**：这是"**可改写后缀**"模式，但修订发生在 **encoder 表示层**（后缀对应的音频区段重算），而非云厂商那种"重发整句文本"。两者都需要"stable_until 之后可变"。

### Zhipu / Z.AI GLM 系（GLM-ASR-2512 / GLM-4-Voice）

来源：Z.AI 文档 https://docs.z.ai/guides/audio/glm-asr-2512 ；GitHub zai-org/GLM-4-Voice（2026-06-05）。

- **GLM-ASR-2512**：REST `POST /api/paas/v4/audio/transcriptions`，**支持 `stream=true`**（OpenAI-compatible 风格）。但传输层（SSE/WS/chunked）、partial 是否改写、时间戳、session 生命周期**文档全缺**——**低确认**，需实测或查更细 API 参考。
- **GLM-4-Voice**：端到端语音对话 LLM，GLM-4-Voice-Tokenizer 把语音→离散 token（12.5 token/s），GLM-4-Voice-Decoder（CosyVoice flow-matching）**支持流式合成**（10 token 即可起播）。ASR 是其内部理解能力，**未暴露为独立流式转写接口**；流式属性主要在 TTS-decoder 侧。

### Microsoft VibeVoice（TTS，证伪）/ VibeVoice-ASR（批）

来源：HF microsoft/VibeVoice-1.5B、VibeVoice-Realtime-0.5B、VibeVoice-ASR ；https://microsoft.github.io/VibeVoice/（2026-06-05）。

- **VibeVoice / VibeVoice-Realtime-0.5B = TTS，不是 ASR**（已证实）。next-token diffusion，7.5Hz 连续 token，长达 90min / 4 说话人；"realtime"指**流式语音合成**（~300ms 首音、interleaved windowed 边收文本边出声）。**与 ASR 标准无关，仅作排除登记。**
- **VibeVoice-ASR（独立模型）**：60min 单趟 long-form，联合 ASR + diarization + timestamp，输出结构化 who/when/what，50+ 语言，支持 hotwords。**整段处理、非流式。**

### Zoom Scribe API（批 / 快同步，无公开增量流）

来源：https://www.zoom.com/en/products/ai-services/scribe-api/（2026-06-05；正文经 302 跳转登录页未取全，依据产品页摘要与开发者博客）。

- 提供 **Fast Sync**（整文件低延迟同步转写）+ **Batch**；宣称"near real-time to batch"。
- **无公开 WebSocket 增量/ partial 协议文档**："near real-time"指快速同步返回，而非"边说边出 partial"。Open ASR Leaderboard #1。标注中确认（streaming 缺失基于现有公开文档）。

### bosonai Higgs Audio（v2 TTS / 3.0 STT）

来源：boson.ai 博客 higgs-audio-v2 / higgs-audio-v3-stt ；HF bosonai/*（2026-06-05）。

- **Higgs Audio v2 = TTS 为主**（Dual-FFN，25Hz token）。其训练管线用到多个 ASR 模型做标注，但 v2 本身是生成。
- **Higgs Audio 3.0 = STT 基础模型**，94 语言，支持有/无语言提示，LibriSpeech clean WER 1.55（优于 Whisper-large-v3 2.10）。**仅见离线 WER 展示，无流式文档**——标注中确认（无流式）。

### WeNet U2++ / Kaldi-Vosk online（经典两遍流式，新增对照）

来源：论文 https://arxiv.org/pdf/2012.05481（U2）、https://arxiv.org/pdf/2203.15455（WeNet 2.0/U2++）、https://arxiv.org/pdf/2211.00941（Fast-U2++）（2026-06-05）。

- **U2/U2++**：单模型统一流式与非流式。**第一遍 CTC prefix beam search 逐 chunk 出低延迟 partial**；**第二遍 attention rescoring** 用 L2R + R2L decoder 对 n-best **重打分**得到定稿。**dynamic chunk training** 让一个模型支持任意 chunk size，调 chunk 即平衡延迟/精度。
- **意义**：经典且广泛部署的"**两遍**"范式 —— 第一遍流式前缀，第二遍对**整段** n-best 重排，可能把第一遍的稳定结果整体改写。这与"逐 token 不可变前沿"冲突，但与"endpoint 后对该段做一次 final 重排/重格式化"高度吻合（见末节）。

## 3. 新流式模式 / 对标准的启示

把本批引擎按"对我们提案模型（a 稳定前缀 frontier + stable_until；b 稳定 segment id；c interim 可改写含 post-final 重格式化；d streaming-input ⟂ streaming-output；e 透明重连）"的压力测试逐条记录。标"✅ 已覆盖 / ⚠️ 需细化 / ❗ 可能无法表达"。

### 3.1 ✅ Delayed Streams Modeling（Kyutai STT、Voxtral Realtime）——逐 token 不可变前沿 + padding token
DSM 把流式输出建模为"与音频同帧率、每 80ms 发一个 token 或 padding token，已发即定（immutable）"。
- 对我们模型：这是 `stable_until` 的**理想退化情形**——`stable_until` 永远等于已发文本末尾，无回退。我们的"稳定前缀 frontier"**天然覆盖**。
- ⚠️ **细化点 1（padding/无文本帧）**：模型会发"这 80ms 没有文本"的 padding 信号。我们的事件模型应允许"音频推进但 segment 文本不变"的心跳/时间推进事件（否则无法表达"模型在等证据"与"静音"的区别，而后者正是 endpoint 依据）。建议 streaming 事件携带一个 `audio_processed_until` / 时间游标，独立于 `stable_until`（文本游标）。
- ⚠️ **细化点 2（时间戳来源）**：word-level timestamp 是帧对齐"免费"产物，不是事后 forced-align。我们的 timestamp 能力描述应区分"原生帧对齐"vs"后处理对齐"vs"无"——精度/可得性不同。
- ⚠️ **细化点 3（delay 是一等参数）**：Voxtral 的 80ms–2.4s、Kyutai 的 0.5s/2.5s 是**模型固有的可配延迟**，直接决定 interim 何时出现。标准的 streaming-output 能力应能声明"固定/可配输出延迟"作为 capability，而非假设"尽快出 partial"。

### 3.2 ❗ encoder-decoder + 流式解码策略（Canary waitk/alignatt）——"逐 chunk 重预测同位置 token"
Canary streaming 下"同一 chunk 内可对同一位置 token 预测多次"，alignatt 还能一个 chunk 发多 token。这**不是单调增长的稳定前缀**，而更像"当前 hypothesis 被反复重算"。
- ❗ **潜在无法表达**：若 provider 在两次事件间**回退已发 token 再重发不同 token**（前缀都变），我们的 `stable_until` 需要能临时收缩到更早位置。提案里 `stable_until` 是"已冻结字数"，**只增不减**才安全；waitk/alignatt 在极端情况下可能要求"未冻结区任意改写"。
- **建议**：明确"`stable_until` 左侧绝对不可变，右侧（含已显示但未冻结部分）可被整体替换"。只要 provider 不冻结过早，我们的"段文本全量 + stable_until"仍可承载（每个事件发当前完整 hypothesis + 一个保守的 stable_until）。**前提**：adapter 必须保守设置 stable_until（宁可偏小），否则会违反不可变承诺。这应写成对 adapter 的硬约束。

### 3.3 ⚠️ Moonshine v2 provisional→finalized（encoder 端覆盖）
后缀对应的 encoder 状态在更多音频到来后被 finalized 覆盖（~320ms 未来上下文后定稿）。
- 这是"未冻结后缀可改写"的另一实例，**与我们 `stable_until` 模型一致**：finalized 边界 = `stable_until`，provisional 后缀 = 未冻结可变区。
- ⚠️ **细化点**：finalize 不是由"标点/静音"触发，而是由"未来上下文长度（帧数）"触发。标准应允许 `stable_until` 推进的**触发原因**多样化（时间/上下文长度 vs VAD/标点），至少作为 metadata，便于应用决定何时"锁定"显示。

### 3.4 ⚠️/❗ 两遍重打分（WeNet U2++、Granite two-pass、Granite 4.1 NLE 编辑）——对"已 final 段"的整体重写
三种形态都在"第一遍/CTC 假设"之后做**一次性整段重写**：
- WeNet：endpoint 后 attention rescoring 重排 n-best → 整段文本可能与流式 partial 不同。
- Granite two-pass / 4.1 NLE：双向 LLM 对 CTC 假设做 copy/insert/delete/replace 编辑（虽是离线，但形态相同）。
- 对我们模型：这正是提案 (c) 的 **"post-'final' reformatting revision，keyed by same segment id"** 想覆盖的场景。✅ **设计方向正确**。
- ⚠️ **细化点 1（编辑而非替换）**：NLE 的输出是**编辑操作序列**（insert/delete/replace at positions），不是"新整段文本"。我们若只支持"用新全文替换该 segment"，会丢失"哪些字被改了"的信息（对 UI 高亮/字幕回滚有用）。建议 revision 事件**可选**携带 diff/edit-ops（增量改写），而不仅是全量替换。
- ❗ **细化点 2（跨段重写 / re-segmentation）**：two-pass rescoring 可能**改变分段边界**（把两个流式段合并、或挪动句界）。我们的"segment id 稳定、修订只针对单个未冻结段"假设在此**可能被打破**——若 final 重排把 segment A、B 合并成 C，A/B 的 id 该如何处置？
  - **建议补强**：引入"段生命周期"事件：除 `revise(id)` 外，需要 `supersede(old_ids[] → new_id)` 或 `merge/split` 语义，让一次 final reformatting 能合并/拆分既有 segment 并保持 id 可追溯。否则"稳定 id"承诺会与"final 重分段"矛盾。这是本批对标准最实质的挑战。

### 3.5 ⚠️ ElevenLabs commit「清空累积段、起新段、不丢上下文」+ committed 是否可再改未明
- "commit 清空 partial 累积、开新段" = 我们的"endpoint → 冻结当前 segment、开下一个 segment id"，✅ 对齐。
- ❗ **缺口**：官方**未承诺** committed 段事后不被改写。若 ElevenLabs 后续仍可修订已 committed 段（很多系统会做尾部标点/数字 ITN 的延迟修正），那就是 (c) post-final revision 的真实用例；若承诺不可变，则 committed = 永久冻结。**这个二选一直接决定我们 segment 的 `frozen` 是否是终态。** 需实测确认。**建议**：标准把"final/committed"分两级——`final`（不再因新音频改写）与 `closed`（连后处理重格式化也不再有）。provider 只能保证到哪一级，应可声明。

### 3.6 ✅ streaming-input ⟂ streaming-output 二分被强力验证
本批清楚地把引擎分成三类，证明二分正交且必要：
- **streaming-input + streaming-output**：Kyutai、Voxtral Realtime、Nemotron Streaming、Canary streaming、ElevenLabs Scribe v2、GLM-ASR(声明)、WeNet U2++。
- **streaming-input + 非增量 output（VAD 切段后离线识别）**：FireRedASR2S（流式 VAD + 离线 AED/LLM）、SenseVoice + 外接 VAD。→ 即"接受连续音频，但只在段末出整段结果"。我们的模型应能表达"streaming-input=yes、streaming-output=仅段末一次（无 interim）"。✅ 二分覆盖，但 ⚠️ 需要一个"**no-interim streaming**"档（每段只发一个 final，无 partial）。
- **纯 batch**：SenseVoice、Cohere、Granite、Omnilingual、VibeVoice-ASR、Higgs 3.0、MiMo-V2.5-ASR（开源）、Voxtral Transcribe V2、Zoom（基本）。

### 3.7 ⚠️ 运行时可切换的延迟档（Nemotron 4 档、Voxtral 连续、Canary 策略）
多个引擎允许**运行时**在延迟–准确率 Pareto 点间切换（Nemotron `[70,r]` 四档、Voxtral delay 参数、Canary waitk vs alignatt）。
- 对标准：streaming session 的"目标延迟 / 右上下文"应是**会话级可配参数**，且**可中途调整**。我们的能力模型若只在"建立连接时"固定延迟，会损失这种灵活性。建议把 `target_latency_ms` / `right_context` 列为标准 streaming 配置项（provider 可忽略）。

### 3.8 透明重连（提案 e）——本批未发现反例，但留意会话上限
- 大多数引擎是本地库或单连接 WS，未暴露强制会话上限（ElevenLabs 有 `session_time_limit_exceeded` 错误，是已知上限）。
- ⚠️ 对 DSM 类**有状态**模型（Kyutai/Voxtral 的 decoder 持续累积 KV/状态）：透明重连要求 adapter 在重连后**恢复 decoder 状态或无缝衔接前缀**——但这些模型状态是 token 级因果状态，跨进程迁移成本高。标准的"重连后 segment id/时间戳/语言连续"承诺，对这类模型实现上更难（可能需要重喂近期音频）。**建议**在 capability 里区分"reconnect: seamless / lossy(需重叠音频) / unsupported"。

### 3.9 证伪 / 排除清单（避免后续误纳）
- **TTS，非 ASR**：Microsoft VibeVoice / VibeVoice-Realtime-0.5B；bosonai Higgs Audio v2；（GLM-4-Voice、MiMo-Audio 含生成，转写非主接口）。
- **确认无流式（批）**：SenseVoiceSmall（论文明言）、Cohere Transcribe、Granite Speech 3.3/4.x、Omnilingual ASR、VibeVoice-ASR、Higgs Audio 3.0、MiMo-V2.5-ASR（开源版）、Voxtral Transcribe V2、Zoom Scribe（公开层面）、FireRedASR(2)-AED/-LLM 的 ASR 模块本身。

## 4. 确认度与待办

- **高确认（官方/论文/模型卡/复刻直证）**：Nemotron Streaming 档位、Canary waitk/alignatt、Kyutai DSM + word ts + semantic VAD、Voxtral Realtime padding token + immutable（antirez 复刻）、SenseVoice 非流式（论文+Issue）、Moonshine v2 provisional/finalized、Cohere 批、Granite 批 + 4.1 NLE 编辑、Omnilingual 批、VibeVoice 是 TTS、WeNet U2++ 两遍。
- **中确认**：Zoom Scribe 无公开增量流；Higgs 3.0 无流式文档；MiMo-V2.5-ASR 开源批（产品宣传流式存疑）。
- **低确认 / 待实测**：
  - GLM-ASR-2512：仅见 `stream=true`，传输层/partial 修订语义/时间戳/会话生命周期全缺。
  - ElevenLabs Scribe v2：committed 段是否会被后续输出再改写（决定 §3.5 二选一）；session 最大时长/chunk 上限数值。
  - Nemotron Streaming / Parakeet-unified：RNNT cache-aware 是否严格 immutable（未逐字确认）。
  - Kyutai WS server 消息格式与显式 EOI 信号（README 未列）。
- **对标准的最高优先级动作项**（按影响排序）：
  1. §3.4 **re-segmentation / 段合并拆分**：增加 `supersede/merge/split` 段生命周期语义——这是"稳定 segment id"在两遍重打分下最可能破裂处。
  2. §3.2 把"`stable_until` 左侧绝对不可变、adapter 必须保守"写成硬约束（应对 Canary 逐 chunk 重预测）。
  3. §3.5 把 final 分级为 `final`（音频意义上定稿）与 `closed`（含后处理重格式化定稿），provider 声明能保证到哪级。
  4. §3.1 streaming 事件增加独立于 `stable_until` 的**音频时间游标**与 padding/心跳，支持 DSM 的"等证据"与时间推进。
  5. §3.3 revision 事件可选携带 **edit-ops/diff**（应对 Granite NLE 式编辑，利于 UI）。
  6. §3.6/§3.7 capability 增加："no-interim streaming"档、`target_latency`/`right_context` 会话级可调、`reconnect: seamless/lossy/unsupported`。
