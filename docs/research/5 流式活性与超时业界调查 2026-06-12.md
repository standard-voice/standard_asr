# 流式活性与超时业界调查（2026-06-12）

**背景**： 指出默认 `done_timeout=30s` / `max_idle=120s` 会误杀合法的
静音直播会话（云 WS 引擎静音期常无任何消息）。在 v1 冻结前需决定：心跳义务（MUST）、
放宽默认、还是标准层注入心跳。本调查对 9 家云端服务 + 3 个开源实现的官方文档逐项核对
五个问题：服务端闲置断连、客户端 keepalive 义务、静音期接收侧行为、会话时长上限、
官方 SDK 客户端行为。三个独立调查代理并行执行，引用均来自官方文档（标注 UNCONFIRMED
者除外）。

## 核心结论（驱动 spec §6.1 / §4.4 的设计）

1. **业界一致把流活性锚定在「音频流」上，没有任何一家锚定在「转写事件流」上。**
   服务端断连条件全部是「客户端停止发音频」（Deepgram 10s / AWS 15s / Google 近实时
   节奏 / ElevenLabs 快速但未公布 / Speechmatics 1h / AssemblyAI 默认不断），无一是
   「服务端停止发结果」。
2. **接收侧静音行为根本性分歧**（这正是 M.1.1 要统一的碎片化）：Deepgram 持续发空
   Results、Azure 定期发 NoMatch（默认 ~15s）、Speechmatics 每块回 AudioAdded ack、
   Vosk 每块都回；而 AssemblyAI / Google / OpenAI / ElevenLabs 完全静默。任何
   「N 秒无事件即判死」的客户端定时器在后一组上必然误杀。
3. **十二个官方 SDK / 实现里零个运行「收不到转写结果 → 中止会话」的看门狗。**
   活性检测一律在传输层（WS ping/pong、gRPC deadline），断连经 error callback 上报；
   自动重连只有 Azure Speech SDK 做（Google 官方 endless-streaming 教学让应用自己
   旋转 5 分钟流；LiveKit / Pipecat 等编排框架被迫自建重连层）。
4. **keepalive 义务在发送侧**：AWS 官方明文「无语音时编码并发送等长静音（PCM 零字节
   流）」；Deepgram / AssemblyAI 提供显式 KeepAlive 文本消息；Pipecat 对 ElevenLabs
   每 5s 注入静音块。与标准层同层的编排框架（Pipecat/LiveKit）做的是发送侧 keepalive +
   重连，不是接收侧事件定时器。

**由此得出的设计**（已落地，spec §6.1）：`done_timeout` 重定位为「管线不活动兜底」——
锚点 = max(最近事件, 最近一次 `audio_chunks` 消费)，默认 300s；适配器消费喂入音频即
证明管线活着，静音会话零义务存活；`end_audio()` 后锚点冻结，退化为「done 必达」窗口。
`max_idle` 默认禁用（opt-in）。心跳维持 MAY + 两种 SHOULD（镜像原生接收侧活动 / 长
静默工作期），标准层 MUST NOT 合成心跳。传输层失败的及时上报为适配器 MUST。应用经
`start_transcription(deadlines=...)` 逐字段覆盖。

## 各家明细

### 云端服务

| 服务 | 服务端断连（无音频） | 客户端 keepalive | 静音期接收侧 | 时长上限 | 官方 SDK 客户端行为 |
|---|---|---|---|---|---|
| Deepgram | 10s 无任何帧 → close 1011 `NET-0001` | `{"type":"KeepAlive"}` 每 3–5s，或送静音音频；KeepAlive 无响应且不能永久维持（至少需一条音频） | 持续发空 transcript 的 Results | 未公布 | Python SDK 可选 `keepalive:true` 自动发；无自动重连（JS 有手动 `reconnect()`）；无事件看门狗 |
| AssemblyAI v3 | 默认**不断**（计费持续）；可选 `inactivity_timeout` 5–3600s | 仅在自设 `inactivity_timeout` 时需要；音频也重置 | **完全静默**（每个静音期至多一条 partial） | 3h（close 3008） | 无看门狗、无自动重连；官方指南教应用自己跟踪 `last_transcript_received` |
| Speechmatics | 1h 无音频 → `idle_timeout`，提前 15/10/5min 发 Warning | 无应用层机制；音频即 keepalive；建议 WS ping/pong 60s | 每个 AddAudio 回 `AudioAdded` ack | 48h（`session_timeout`） | SDK 暴露 ping_interval/ping_timeout（传输层）；重连留给应用（README 示例用 tenacity） |
| Google STT v1/v2 | 音频须近实时送达，停滞即 "Timeout Error: Long duration elapsed without audio"；可选 VAD `speech_start/end_timeout` 关流 | 持续送音频（含静音）；无显式机制 | 默认无输出；v2 可开 `SPEECH_ACTIVITY_BEGIN/END` 事件 | **~5min/流**（v1、v2 同） | gRPC deadline（.NET 5000s）；无看门狗；官方 endless-streaming 教学 = 应用自己旋转流并桥接缓冲 |
| AWS Transcribe | **15s 无音频** → `BadRequestException`（数值来自错误消息本身 + re:Post，dev guide 未写明） | 官方最佳实践：「无语音时编码发送等长静音（PCM 零字节）」 | 未文档化（推测安静） | 4h 硬上限（`LimitExceededException`）；`SessionResumeWindow` 1–300min 可续 | SDK 上抛服务异常，应用自启新流；无看门狗、无自动重连 |
| Azure Speech | 连续模式无固定值（单次模式 `InitialSilenceTimeout`） | 无（SDK 自管连线） | **定期发 NoMatch 空结果**（initial-silence 属性控制，默认 15000ms） | 实时 STT 无通用上限（diarization 240min） | **SDK 自动重连**（Connection 类文档明言）；失败经 `Canceled` 事件；无看门狗 |
| OpenAI Realtime | 无文档值；实测有裸 1006 断连报告；会话到期发 `error: session_expired` | 无机制；持续 `input_audio_buffer.append`（含静音） | server_vad 只在语音边界发事件；持续静音**零事件**；可选 `idle_timeout_ms` 发 `timeout_triggered`（不断连） | 60min（Azure 版 30min） | Agents SDK 无心跳/重连逻辑；LiveKit 自建重连（issues #2341/#3145） |
| ElevenLabs Scribe RT | 闲置断连快（数值未公布；TTS 是 20s 可调到 180s，STT 未文档化）；有 `insufficient_audio_activity` 错误 | 无原生机制；Pipecat 每 5s 注入静音块作 keepalive + 自动重连 | 安静；VAD commit 策略下静音触发 `committed_transcript` | 有上限（`session_time_limit_exceeded`），数值未公布 | 官方 JS SDK：无 keepalive、无看门狗、无自动重连；错误经统一 ERROR 事件 |

### 开源 / 本地

| 实现 | 行为 |
|---|---|
| Vosk server | 无超时、无上限；每个音频块都回（含空/不变 partial）；`{"eof":1}` 结束 |
| whisper-streaming (UFAL) | 裸 TCP demo；无超时/keepalive/上限 |
| WhisperLive (Collabora) | 唯一有硬上限的 OSS：`--max_connection_time` 默认 600s；无闲置超时 |
| sherpa-onnx online WS | 无任何超时/keepalive/时长旗标；端点检测纯靠 trailing-silence 规则 |

### OSS 共同惯例

连接活到客户端不想要为止；「持续送音频（含静音）」是唯一 keepalive；显式 EOF 哨兵
结束；活性留给 WS 库默认（如 Python `websockets` ping_interval=20s/ping_timeout=20s）。

## 对三个原始方案的裁定

- **方案 1（MUST 心跳）**：否。会强加一个**业界不存在对应物**的接收侧义务——对
  Deepgram/Azure/Speechmatics 形态是免费翻译，但对 AssemblyAI/Google/OpenAI/
  ElevenLabs/几乎全部本地引擎，适配器只能写盲定时器（只证明适配器协程活着，等于
  每个作者各自实现一遍方案 3）。违 P.1.2。心跳的合理部分降级为两种 SHOULD 保留。
- **方案 2（放宽默认值）**：方向对但不充分——300s 只是把误杀引信拉长十倍（会议
  静音 >5min 完全正常）。被「活动锚定」方案取代：消费喂入音频即活性证据，静音
  存活不再依赖任何超时数值的宽容度。
- **方案 3（标准层注入心跳）**：否。基类定时器只能证明基类活着（WS 死了照发），
  既掏空挂死防护又伪造引擎未产出的事件。spec §4.4 现明文：标准层 MUST NOT 合成心跳。

**最终设计 = 方案 2 的重定位 + 音频消费锚定**（交付方案 1 的结果、方案 2 的零义务、
不碰方案 3 的伪造），详见 spec §6.1「活性与终止保证」。
